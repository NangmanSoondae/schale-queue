## T2. M8 실측 — SSE 직렬 브로드캐스트 (핵심)

> 리뷰 백로그 M8: "SSE 브로드캐스트가 단일 스케줄러 직렬 순회 — 부하 실측 후 결정".
> 측정법: 구독 N개를 열고 **같은 tick 의 position 이벤트 도착 스프레드(최초~최후)** ≈ 직렬 브로드캐스트 1회 소요.

### (1) 수정 전 — 구독자당 개별 왕복(ZRANK+EXISTS+ZCARD ×N 직렬)

| 구독 수 | tick 1회 소요(p50) | 폴링 주기(1s) 대비 |
|---------|--------------------|---------------------|
| 500 | ~370ms | 37% |
| 1,000 | ~730ms | 73% |
| 2,000 | **~1,490ms** | **149% — 주기 초과(포화)** |

구독자당 **~0.74ms 로 선형 증가** — 지배 항은 emitter send 가 아니라 **구독자별 Redis 왕복 RTT**(컨테이너 NAT 경유 ×2회). 2,000 구독이면 순번 갱신·입장 알림이 1.5s+ 지연 → **M8 수정 필요 판정**.

### (2) 수정 — 상품별 벌크 조회로 왕복을 상수화

tick 마다 상품(goodsId)별로 **토큰 `MGET` 1회 + 순번 파이프라인(`executePipelined` ZRANK×N) 1회 + `ZCARD` 1회**만 왕복하고, emitter 순회에는 인메모리 send 만 남긴다. (`QueueService.getPositions` / `AdmissionTokenService.membersWithValidToken` 신설, `QueueStreamService.broadcast` 벌크 경로 전환)

### (3) 수정 후

| 구독 수 | tick 1회 소요(p50) | 개선 배율 |
|---------|--------------------|-----------|
| 2,000 | **~120ms** (p90 191ms) | **12.4×** |

- 모든 tick 이 주기 내 완주(13/13 full tick, 이전엔 15s 에 6 tick). 구독자당 비용 0.74ms → **~0.06ms**.
- 선형 외삽 시 **~8,000 구독까지 1s 주기 내 처리 가능**(단일 인스턴스). 단, 외삽이다 — 아래 한계 참조.
- ✅ **M8 종결**: 실측 근거로 수정·재실측 완료.

### N=5,000 직접 실측 실패 (정직 기록)

5,000 동시 구독 시도는 **하네스 한계**로 무산: 벤치 반복으로 Windows TIME_WAIT 소켓 1만+ 누적 → 클라이언트 임시 포트 고갈 → 연결 자체가 ~2,000 에서 정체/0건. 애플리케이션이 아닌 **부하기(단일 Windows 머신) 제약**이다. 5,000+ 구독 실측은 리눅스 원격 부하기(또는 다중 부하기)로 후속 수행한다.

## T3. E2E 풀플로우 무유실 — S8 (Phase 4 이후 첫 부하 검증)

시나리오: 회원 1,000명 백로그 → 워커 입장(배치 스로틀) → **주문 + 결제 확정**(k6 `order-pay.js`, 100VU) → 아웃박스 → Kafka `order.completed` → 정산 컨슈머.

| 검증 | 결과 | 판정 |
|------|------|------|
| 플로우 완주 | 1,000/1,000 (191 flow/s) | ✅ |
| S7 5xx | 0건 (2,000 req) | ✅ |
| **S8 무유실**: 완료 주문 = PAID 결제 = 정산 원장 | **1,000 = 1,000 = 1,000** | ✅ |
| 정산 수렴 시간 (k6 종료 후 컨슈머 lag) | **~2s** | ✅ |
| 재고 정합: available+reserved+sold=total, sold=완료주문 | 4,000+0+1,000=5,000, sold=1,000 | ✅ |
| 주문 p95 / 결제확정 p99 | 905ms / <500ms(threshold 통과) | 참고 |

**아웃박스 → Kafka → 정산 파이프라인이 부하 하에서 단 1건도 새지 않음을 처음으로 실측 입증**(S8). Phase 4 설계(트랜잭셔널 아웃박스 + 멱등 컨슈머)의 완결 증거.

## 부하테스트가 찾아낸 결함 (수정 완료)

1. **SSE 대량 이탈 → ERROR 로그 폭탄** ([troubleshooting No.13](./troubleshooting.md)): `AsyncRequestNotUsableException`(클라이언트 끊김)을 500 catch-all 이 시스템 오류로 오분류. 전용 핸들러로 debug 강등.
2. **M8 브로드캐스트 포화** (상기 T2): 백로그 항목이 실측으로 확정 → 벌크 경로로 수정.

## 결론

- **컨테이너 풀스택에서도 핵심 SLO(S1/S2/S7) 충족** — 정상상태 성능은 호스트와 동등, 컨테이너화 비용은 버스트 NAT 경로에 국한(개발 장비 특성, 프로덕션 재측정 대상).
- **M8 실측 → 수정 → 12.4× 개선 재실측**으로 백로그 종결. 단일 인스턴스 SSE 수용량 ~2,000 → 외삽 ~8,000.
- **S8 무유실 최초 실측 통과** — Kafka 후방 파이프라인(아웃박스·멱등·정산)이 부하 하에서 0 유실.

## 후속 (범위 밖)

- 5,000+ SSE 구독 실측(리눅스/원격 부하기), N회 반복 평균.
- 프로덕션 유사 네트워크(리눅스 브리지)에서 스파이크 재측정.
- SSE 폴링 모델 자체의 이벤트 기반 전환(Pub/Sub push)은 여전히 선택지 — 단, 벌크 경로로 단일 인스턴스 목표 수용량을 확보했으므로 스케일아웃 요구가 생길 때 재평가.
