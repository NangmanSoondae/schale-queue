# ADR-001. 핵심 엔티티 설계 — 트래픽 병목과 락 경합을 분리하는 테이블 전략

> **ADR(Architecture Decision Record)**: 본 문서는 Phase 1 데이터베이스 설계 과정에서 내린 핵심 결정과 그 근거, 그리고 감수해야 할 트레이드오프를 기록한다.

- **상태(Status)**: Accepted
- **일자**: 2026-06-19
- **관련 Phase**: Phase 1 (기반 구축 — ERD 설계)
- **결정 대상**: Member, Goods, Stock, Order, OrderItem, Payment 6개 엔티티의 분리 전략

---

## 1. Context (상황)

**Schale Queue**는 일반적인 커머스가 아니라 **대규모 선착순(first-come, first-served) 트래픽**을 전제로 한다. 이 환경에서 테이블 설계는 단순한 데이터 정규화의 문제가 아니라, **시스템의 생존을 좌우하는 동시성 설계 그 자체**다.

선착순 오픈 순간의 트래픽은 두 가지 상반된 성격의 부하를 동시에 발생시킨다.

| 부하 유형 | 성격 | 발생 지점 | 위험 |
| :--- | :--- | :--- | :--- |
| **대량 읽기(Read)** | 상품 정보 조회. 수만 명이 동일 상품을 동시에 본다. | 상품 상세 페이지 | DB 부하 증가, 캐시 미스 |
| **집중 쓰기(Write)** | 재고 차감. 단 하나의 행(row)에 쓰기 경합이 집중된다. | 재고 차감 | **행 단위 락(Row Lock) 경합**, 응답 지연, 데드락 |

만약 상품 정보와 재고 수량을 **하나의 테이블에 둔다면**, 재고를 차감하기 위해 거는 쓰기 락이 **상품 정보를 단순 조회하려는 수만 건의 읽기 요청까지 직간접적으로 간섭**하게 된다. 즉, "변하지 않는 정보(상품명, 설명)"와 "초당 수천 번 변하는 정보(재고)"가 같은 락 단위에 묶이는 **거짓 공유(false sharing)** 문제가 발생한다.

따라서 우리의 핵심 과제는 명확하다:

> **"읽기 부하"와 "쓰기 경합"을 물리적으로 분리하여, 서로가 서로를 방해하지 않게 만든다.**

---

## 2. Decision (결정)

### 결정 1. Goods(상품)와 Stock(재고)을 1:1로 분리한다

상품의 **불변에 가까운 메타 정보(Goods)**와 **고빈도로 변경되는 재고 수량(Stock)**을 별도 테이블로 분리한다.

- **Goods**: name, description, price, openAt 등 **읽기 위주(read-heavy)**, 변경 빈도가 낮은 데이터.
- **Stock**: totalQuantity, remainQuantity 등 **쓰기 경합(write-contended)**이 집중되는 데이터.

> 🔄 **후속 refine (2026-06-24)**:
> - Stock의 차감 모델과 카운터 스키마는 [`12_adr_004_stock_reservation.md`](./12_adr_004_stock_reservation.md)에서 **예약 기반 3-카운터(`total` + `available`/`reserved`/`sold`)**로 확장되었다.
> - Goods에는 **상품별 운영 설정** `maxPurchasePerMember`(1인 한도, 기본 1)·`paymentTimeoutMinutes`(결제창, 기본 10·1~30분)가 추가된다(변경 적은 설정 메타정보 → Stock이 아닌 Goods). 근거: [`11_domain_policy.md`](./11_domain_policy.md) P-O2·P-O3.
> - 본 ADR-001의 1:1 분리·단일 행 락 원칙은 그대로 유효하다.

이렇게 분리하면, 재고 차감을 위한 비관적 락(`SELECT ... FOR UPDATE`)이 **오직 Stock 테이블의 단일 행에만 걸린다.** 상품 정보 조회는 락과 무관하게 자유롭게(그리고 캐시 가능하게) 수행된다. 락의 영향 범위(blast radius)를 최소화하는 것이 핵심이다.

### 결정 2. Payment(결제)를 Order(주문)에서 1:1로 분리한다

결제는 **외부 PG사 연동**이라는 본질적으로 불안정하고 느린(I/O 바운드, 외부 의존) 과정을 포함하며, **실패·타임아웃·재시도**가 일상적으로 발생하는 영역이다. 이를 Order 본문에 합치지 않고 분리한다.

- **Order**: 주문의 **확정된 사실**(누가, 무엇을, 얼마에 주문했는가)을 담는다. 상대적으로 안정적이다.
- **Payment**: PG 승인번호(paymentUid), 결제 상태(status), **결제 만료 시간(timeoutAt)** 등 **변동성이 크고 생명주기가 복잡한** 결제 트랜잭션 상태를 담는다.

`timeoutAt`을 Payment에 두면, 별도의 워커(`module-worker`)가 **만료된 결제만 효율적으로 조회·정리**할 수 있다. 결제 상태 변경이 잦더라도 그 쓰기 부하가 Order 테이블로 전파되지 않는다.

### 결정 3. 연관 관계는 객체 참조가 아닌 ID 참조(Long)로 맺는다

`@ManyToOne` 같은 JPA 객체 연관 대신, `goodsId`, `memberId`, `orderId`처럼 **식별자(FK) 값을 직접 보유**한다.

- **애그리거트 경계 보존**: 주문이 상품 객체 전체를 끌어오지 않는다. 각 엔티티는 독립적으로 로딩·관리된다.
- **락 전파 차단**: cascade 및 지연 로딩으로 인해 의도치 않은 추가 락/쿼리가 발생하는 것을 방지한다.
- **명시적 제어**: 어떤 데이터를, 언제, 어떤 락으로 가져올지 개발자가 명시적으로 결정한다.

## 3. Consequences (결과 및 트레이드오프)

### ✅ 이점 (동시성 / 락 경합 관점)

1. **락 경합의 국소화**: 재고 차감 락이 Stock의 단일 행에만 걸리므로, 읽기 트래픽과 충돌하지 않는다. Phase 2의 비관적 락(`FOR UPDATE`) 전략의 효과를 극대화하는 토대가 된다.
2. **캐시 친화성**: 변경이 적은 Goods는 Redis 등으로 캐싱하기에 이상적이다. 재고(Stock)는 캐시에서 분리해 정합성을 DB/분산락으로 엄격히 관리할 수 있다 — **변하는 것과 변하지 않는 것의 분리.**
3. **결제 실패의 격리**: PG 타임아웃·실패가 주문 데이터의 안정성을 해치지 않는다. `timeoutAt` 인덱스로 만료 결제를 워커가 배치 처리하기 용이하다.
4. **수평 확장 여지**: ID 참조 기반 애그리거트 분리는, 향후 도메인별 DB 분리(혹은 샤딩)로 나아갈 때의 마찰을 줄인다.

### ⚠️ 트레이드오프 및 향후 관리 포인트

1. **조인(JOIN) 비용 증가**: 테이블이 분리된 만큼, 상품+재고를 함께 보려면 추가 조회가 필요하다. → 읽기 화면은 **별도 조회 또는 캐싱**으로 대응하며, 이는 의도된 비용이다.
2. **정합성 책임의 이전(애플리케이션 부담)**: ID 참조는 JPA의 객체 그래프 보장을 포기하는 대신, 참조 무결성과 트랜잭션 경계를 **애플리케이션 코드가 명시적으로 책임**져야 한다. (DDL의 FK 제약으로 최후의 안전망은 유지)
3. **분산 환경의 락은 별도 과제**: 행 단위 락 국소화는 **단일 DB** 기준의 이점이다. Scale-out 시에는 Phase 3의 **Redis 분산 락**이 추가로 필요하다. (ADR-001의 범위 밖, 추후 ADR로 다룬다.)
4. **낙관적 락(@Version) 도입 여부 — 보류**: Stock에 버전 컬럼을 두는 낙관적 락도 후보였으나, 선착순처럼 경합이 극심한(high-contention) 시나리오에서는 충돌·재시도가 폭증하여 비관적 락이 유리할 수 있다. **Phase 2에서 두 전략을 실측 비교한 뒤 결정**한다. 본 Phase에서는 컬럼을 추가하지 않는다.
5. **FK 제약의 쓰기 오버헤드**: 외래키 제약은 무결성을 보장하지만 INSERT 시 부모 행에 대한 점검 락을 유발한다. 극한 부하에서 병목이 확인되면, FK 제약을 애플리케이션 검증으로 대체하는 것을 **별도 ADR로 재검토**할 수 있다.

## 4. 결정 요약

| 결정 | 한 줄 요약 |
| :--- | :--- |
| Goods ↔ Stock 1:1 분리 | **읽기 부하**와 **쓰기 경합**을 물리적으로 격리 |
| Order ↔ Payment 1:1 분리 | 불안정한 **외부 결제**의 변동성을 주문에서 격리 |
| ID 참조(Long) 연관 | 애그리거트 경계 보존 + 락 전파 차단 |
| @Version 보류 | Phase 2 실측 후 락 전략 확정 |
