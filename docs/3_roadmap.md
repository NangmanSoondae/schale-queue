# 3. 전체 개발 로드맵

> **Schale Queue**는 5개의 Phase로 점진적으로 완성된다. 각 Phase는 명확한 산출물과 검증 기준을 가지며, 이전 단계의 한계를 마주한 뒤 다음 단계로 나아간다.
> **v1(Phase 1~5)은 2026-07-07 완주**했고, 이후는 [v2 로드맵](#v2-로드맵-배포-확장--모바일-reach--scale)으로 이어진다.

## 전체 흐름 개요

```
[v1] Phase 1 ──▶ Phase 2 ──▶ Phase 3 ──▶ Phase 4 ──▶ Phase 5          ✅ 완주 (2026-07-07)
       뼈대        핵심        대기열       Kafka·AI     배포·폴리싱
       세팅       비즈니스     시스템       어드민

[v2] v2-a ──────▶ v2-b ────────────▶ v2-c
     GHCR 퍼블리시  클라우드 상시 데모     모바일 앱(푸시)
     (누구나 실행)  (+리눅스 부하 재실측)   (Capacitor·FCM·APK)
```

---

## Phase 1. 기반 구축 (Foundation)

> **"단단한 뼈대 위에 단단한 집을 짓는다."**

- Monorepo 워크스페이스 및 Spring Boot **멀티 모듈**(`module-core` / `module-api` / `module-worker`) 뼈대 세팅
- 공통 인프라 구성: Docker Compose 기반 **MariaDB / Redis** 로컬 환경
- 핵심 도메인 **ERD 설계** (User, Product, Stock, Order, OrderItem 등)
- 전역 설정, 예외 처리, 응답 포맷 등 공통 토대 정비

**산출물**: 빌드 가능한 멀티 모듈 프로젝트, 실행 가능한 로컬 인프라, 확정된 ERD

---

## Phase 2. 핵심 비즈니스 로직 & 동시성 제어 (Core Logic)

> **"먼저 정확함을 증명하고, 그다음 빠름을 추구한다."**

- 상품(Product) 및 주문(Order) 핵심 비즈니스 로직 구현
- 재고 차감 로직 구현 및 **JPA 락(비관적/낙관적 락)**을 활용한 동시성 제어
- **동시성 테스트 작성**: 다중 스레드 환경에서 재고 초과 판매(oversell)가 발생하지 않음을 검증
- 락 전략별 성능/정합성 트레이드오프 측정 및 기록

**산출물**: 정합성이 검증된 주문 로직, 동시성 테스트 코드, 락 전략 비교 분석

---

## Phase 3. 대용량 트래픽 방어 (Queue System)

> **"문(門)을 만들어, 들어올 사람의 수를 통제한다."**
>
> ✅ **완료 (2026-06-26)** — ①② 대기열 골격/도메인 · ③ Worker Consumer · ④ 주문 입장 게이트 · ⑤ 진입 API+SSE · ⑥ 부하테스트·SLO 검증. 핵심 SLO(S1/S2/S4/S5/S6/S7) 실측 충족, 오버셀 0건 증명. 상세: [`load_test_report.md`](./load_test_report.md).

- **Redis Sorted Set** 기반 트래픽 대기열 시스템 도입
- 대기열 진입(enqueue) / 입장(dequeue) / 순번 조회 API 구현
- `module-worker`의 Consumer가 정해진 처리량만큼 대기열을 소비하여 트래픽 평탄화
- **SSE(Server-Sent Events)** 연동: 사용자에게 실시간 대기 순번 및 입장 알림 전송
- Java 21 **Virtual Threads** 적용 및 부하 테스트를 통한 효과 검증

**산출물**: 동작하는 대기열 시스템, 실시간 SSE 연동, 부하 테스트 리포트

---

## Phase 4. 비동기 이벤트 처리 & AI 어드민 (Event-Driven & Intelligence)

> **"사용자 응답 경로에서 무거운 일을 걷어내고, 시스템은 이벤트로 느슨하게 협력하며, 운영에는 지능을 더한다."**
>
> ✅ **완료 (2026-07-01, EDA 범위)** — 트랜잭셔널 아웃박스(ADR-007) · 멱등 컨슈머 · Flyway(ADR-008) · 취소 이벤트 · 정산 컨슈머(#21) · 무유실 방어(ErrorHandlingDeserializer+DLT, #23). 무유실은 Phase 5 부하에서 실측 입증(S8).
> ✅ **AI 어드민(MCP) 완료 (2026-07-06)** — `module-admin-mcp`(Spring AI MCP 서버, stdio) 로 자연어 운영 인터페이스 구현. 읽기 툴 5종(상품/재고/대기열/주문/정산), 쓰기 툴은 write-enabled 게이트로 2차. 근거: [`17_adr_009_ai_admin_mcp.md`](./17_adr_009_ai_admin_mcp.md).

- **Kafka 도입 및 이벤트 기반 아키텍처(Event-Driven Architecture) 전환** — 대기열을 통과한 주문/결제를 Kafka 토픽으로 발행하고, 주문·결제·알림·정산 컨슈머가 구독하여 **비동기·무유실**로 처리한다. (근거: [`7_adr_002_queue_architecture.md`](./7_adr_002_queue_architecture.md))
  - 후방 처리를 동기 호출에서 **이벤트 발행/구독(Pub/Sub)**으로 분리 → 서비스 간 결합도↓, 독립 확장
  - 결제 실패/타임아웃의 재처리·재시도를 오프셋 기반으로 견고하게 설계
- 주문 완료/실패 등 **비동기 알림 처리** 파이프라인 구축 (Kafka 컨슈머로 구현) — 발송은 공용 알림 게이트웨이로 분리 예정 ([`8_adr_003_notification_externalization.md`](./8_adr_003_notification_externalization.md))
- 사용자 응답과 분리된 백그라운드 작업 흐름 정립
- **MCP(Model Context Protocol)**를 활용한 **AI 어드민 제어 환경** 구축 — 자연어 기반 상품/재고/주문 운영 및 모니터링 실험

**산출물**: Kafka 기반 이벤트 파이프라인, 비동기 알림 시스템, MCP 기반 AI 어드민 인터페이스

## v2 로드맵: 배포 확장 & 모바일 (Reach & Scale)

> **"내 노트북에서만 돌던 시스템을, 누구나 만지고 주머니에 넣을 수 있게."**
>
> v1이 "만들 수 있는가"의 증명이었다면, v2는 **도달 범위(reach)**의 확장이다. 각 트랙은 앞 트랙을 선행 조건으로 갖는다: 이미지가 있어야 클라우드에 배포하고, 공개 서버가 있어야 앱이 붙는다.
> 킥오프 결정: 2026-07-07 (퍼블릭 전환 완료 직후).

### v2-a. GHCR 이미지 퍼블리시 — "클론·빌드 없이 실행"

- 릴리스 태그(`v*`) 트리거로 **api / worker / frontend 3종 이미지**를 GitHub Container Registry에 자동 발행
- GHCR 이미지를 그대로 당겨 쓰는 compose 실행 경로 제공 → 누구나 `docker compose up` 한 줄로 전체 시스템 재현
- README에 Quick Start(레지스트리 경로) 안내

**산출물**: `ghcr.io/nangmansoondae/schale-queue-{api,worker,frontend}` 이미지, 릴리스 자동화 워크플로
**검증 질문**: "소스 빌드 없이 이미지만으로 풀스택이 뜨는가?"

### v2-b. 클라우드 상시 데모 — "URL 하나로 보여주는 포트폴리오"

- 클라우드 VM에 GHCR 이미지 기반 풀스택 배포, 상시 접근 가능한 데모 URL 확보
- HTTPS·도메인·기동 자동화(장애 시 자가 복구) 정비
- **리눅스 부하 재실측** 겸용: Windows 환경 한계(TIME_WAIT 포트 고갈)로 미뤄둔 스파이크 NAT·**5,000+ SSE 동시 구독** 실측
- 업체·스펙·비용은 착수 시점에 결정

**산출물**: 공개 데모 URL, 리눅스 부하 리포트(5,000+ SSE)
**검증 질문**: "링크 하나로 대기열→주문 플로우를 체험시킬 수 있는가?"

### v2-c. 모바일 앱 — "대기열 알림을 주머니로" (Capacitor + FCM + APK)

- 기존 React 프론트를 **Capacitor**로 래핑해 안드로이드 앱化 — 코드 재사용, 스토어 심사 없이 **서명된 APK를 GitHub Releases로 직배포**
- **FCM 푸시 연동**: 백엔드에 디바이스 토큰 등록 API 신설, 워커가 대기열 입장(admitted) 시 FCM 발송 → **앱을 꺼놔도 "입장 순서가 됐습니다" 알림** (SSE는 페이지를 열어둔 동안만 유효하다는 v1 한계의 해소)
- 기존 이벤트 파이프라인(아웃박스→Kafka 컨슈머) 위에 푸시 발송 컨슈머를 얹는 구조 — Phase 4 자산의 재활용
- Firebase 프로젝트(무료) 서비스 계정 키는 §5.3.2 시크릿 관리 준수

**산출물**: GitHub Releases의 설치형 APK, FCM 푸시 파이프라인, 디바이스 토큰 API
**검증 질문**: "폰에 설치하고 앱을 닫아도, 내 차례가 오면 알림이 오는가?"

### v2 백로그 (트랙 사이 선별 착수)

- **스케일아웃**: OutboxRelay SKIP LOCKED, SSE 브로드캐스트 Redis Pub/Sub화(다중 api 인스턴스), 컨슈머 그룹 분리
- **MCP 원격화**: stdio → SSE 전송 + 인증 (클라우드 데모와 결합 시 원격 AI 어드민)
- 전체 리뷰 잔여 LOW 선별 (상태머신·큐 REST 시맨틱 등)
