# 3. 전체 개발 로드맵

> **Schale Queue**는 5개의 Phase로 점진적으로 완성된다. 각 Phase는 명확한 산출물과 검증 기준을 가지며, 이전 단계의 한계를 마주한 뒤 다음 단계로 나아간다.

## 전체 흐름 개요

```
Phase 1 ──▶ Phase 2 ──▶ Phase 3 ──▶ Phase 4 ──▶ Phase 5
  뼈대        핵심        대기열       Kafka·AI     배포·폴리싱
  세팅       비즈니스     시스템       어드민
```

---

## Phase 1. 기반 구축 (Foundation)

> **"단단한 뼈대 위에 단단한 집을 짓는다."**

- Monorepo 워크스페이스 및 Spring Boot **멀티 모듈**(`module-core` / `module-api` / `module-worker`) 뼈대 세팅
- 공통 인프라 구성: Docker Compose 기반 **MariaDB / Redis** 로컬 환경
- 핵심 도메인 **ERD 설계** (User, Product, Stock, Order, OrderItem 등)
- 전역 설정, 예외 처리, 응답 포맷 등 공통 토대 정비

**산출물**: 빌드 가능한 멀티 모듈 프로젝트, 실행 가능한 로컬 인프라, 확정된 ERD

---

## Phase 2. 핵심 비즈니스 로직 & 동시성 제어 (Core Logic)

> **"먼저 정확함을 증명하고, 그다음 빠름을 추구한다."**

- 상품(Product) 및 주문(Order) 핵심 비즈니스 로직 구현
- 재고 차감 로직 구현 및 **JPA 락(비관적/낙관적 락)**을 활용한 동시성 제어
- **동시성 테스트 작성**: 다중 스레드 환경에서 재고 초과 판매(oversell)가 발생하지 않음을 검증
- 락 전략별 성능/정합성 트레이드오프 측정 및 기록

**산출물**: 정합성이 검증된 주문 로직, 동시성 테스트 코드, 락 전략 비교 분석

---

## Phase 3. 대용량 트래픽 방어 (Queue System)

> **"문(門)을 만들어, 들어올 사람의 수를 통제한다."**

- **Redis Sorted Set** 기반 트래픽 대기열 시스템 도입
- 대기열 진입(enqueue) / 입장(dequeue) / 순번 조회 API 구현
- `module-worker`의 Consumer가 정해진 처리량만큼 대기열을 소비하여 트래픽 평탄화
- **SSE(Server-Sent Events)** 연동: 사용자에게 실시간 대기 순번 및 입장 알림 전송
- Java 21 **Virtual Threads** 적용 및 부하 테스트를 통한 효과 검증

**산출물**: 동작하는 대기열 시스템, 실시간 SSE 연동, 부하 테스트 리포트

---

## Phase 4. 비동기 이벤트 처리 & AI 어드민 (Event-Driven & Intelligence)

> **"사용자 응답 경로에서 무거운 일을 걷어내고, 시스템은 이벤트로 느슨하게 협력하며, 운영에는 지능을 더한다."**

- **Kafka 도입 및 이벤트 기반 아키텍처(Event-Driven Architecture) 전환** — 대기열을 통과한 주문/결제를 Kafka 토픽으로 발행하고, 주문·결제·알림·정산 컨슈머가 구독하여 **비동기·무유실**로 처리한다. (근거: [`7_adr_002_queue_architecture.md`](./7_adr_002_queue_architecture.md))
  - 후방 처리를 동기 호출에서 **이벤트 발행/구독(Pub/Sub)**으로 분리 → 서비스 간 결합도↓, 독립 확장
  - 결제 실패/타임아웃의 재처리·재시도를 오프셋 기반으로 견고하게 설계
- 주문 완료/실패 등 **비동기 알림 처리** 파이프라인 구축 (Kafka 컨슈머로 구현) — 발송은 공용 알림 게이트웨이로 분리 예정 ([`8_adr_003_notification_externalization.md`](./8_adr_003_notification_externalization.md))
- 사용자 응답과 분리된 백그라운드 작업 흐름 정립
- **MCP(Model Context Protocol)**를 활용한 **AI 어드민 제어 환경** 구축 — 자연어 기반 상품/재고/주문 운영 및 모니터링 실험

**산출물**: Kafka 기반 이벤트 파이프라인, 비동기 알림 시스템, MCP 기반 AI 어드민 인터페이스

---

## Phase 5. 통합 & 배포 (Integration & Polishing)

> **"보이지 않던 백엔드를, 손에 잡히는 제품으로."**

- **React 프론트엔드** 연동: 대기열 화면, 상품 목록, 주문 플로우
- **Docker** 컨테이너 기반 통합 배포 환경 구성
- 전체 시스템 통합 부하 테스트 및 최종 성능 튜닝
- **포트폴리오 폴리싱**: 문서 정비, 트러블슈팅 일지 정리, README 완성

**산출물**: 동작하는 풀스택 데모, 컨테이너 배포 환경, 완성된 포트폴리오
