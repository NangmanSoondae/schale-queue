# ADR-006. 환불·취소 정책 (유형별 · 템플릿화 · 부분환불 · 취소표 쿨다운)

> **ADR(Architecture Decision Record)**: 상품 유형별 환불·취소 정책, 환불정책 템플릿화, 부분환불 도입, 취소표 쿨다운 복원, 보상환불 하이브리드, 결제 재시도 정책을 확정한다.

- **상태(Status)**: Accepted
- **일자**: 2026-06-24
- **관련 Phase**: 설계 결정 (⚠️ **구현은 후순위** — 핵심 큐·동시성 이후)
- **결정 대상**: 환불(refund)·취소(cancel)·재시도(retry)의 도메인 정책
- **관계**: 유형 분리는 [`13_adr_005`](./13_adr_005_bounded_context_split.md) / 재고 모델은 [`12_adr_004`](./12_adr_004_stock_reservation.md) **refine**(cooldown 상태 추가) / 정책 ID는 [`11_domain_policy.md`](./11_domain_policy.md)

---

## 1. Context (상황)

기존 정책에는 환불·취소가 비어 있었고, 선착순 한정 상품 특성상 **되팔이(scalping) 방지**와 **유형별 상이한 환불 규칙**이 필요해졌다. 또한 결제 전 예약 해제(`P-O2`)와 결제 **후** 환불은 시점·돈 흐름·재고 처리가 모두 다르다.

> 구분: **P-O2(예약 해제)** = 결제 *전* PENDING 만료, 돈 안 오감, 재고만 복원. **환불** = 결제 *후* PAID 상태, 실제 환급 + 재고 처리.

---

## 2. Decision (결정)

### 결정 1. 환불 정책은 상품 유형별로 분기한다 (ADR-005)

#### (a) 예매형(TICKET) — 도래일 기준 시간차 부분환불
`GoodsTicketDetail.eventDate`(도래일)까지 남은 기간으로 환불 비율을 차등한다. 비율은 **환불정책 템플릿**으로 정의(결정 2).

| 기준(도래일까지) | 환불 비율(표준 템플릿 예시) |
| :--- | :--- |
| 15일 전까지 | 100% |
| 7일 전까지 | 50% |
| 3일 전까지 | 20% |
| 그 이후 | 0% (환불 없음) |

- 계산 시점 = **취소 요청 시각**, 기준 시간대 **KST 고정**, 경계는 템플릿에 명시(inclusive/exclusive).

#### (b) 배송형(SHIPPING) — 구매확정 기준
- 결제 후 ~ **구매확정 전**(배송 전/중/완료 직후): 변심(청약철회) 환불 **가능**.
- **구매확정 후**(사용자 직접 확정 또는 `autoConfirmDays` 경과): 변심 환불 **불가**(하자/계약불일치는 별도).
- `autoConfirmDays`는 **최소 7일, 7 이상만 설정 가능**(기본 7) — 법정 청약철회권(7일)을 자동확정이 침범하지 않도록 보수적으로 설정.

### 결정 2. 환불 규칙을 재사용 가능한 템플릿으로 둔다

`RefundPolicyTemplate`(예: "공연 표준 15/7/3")을 정의하고 `GoodsTicketDetail.refundPolicyTemplateId`로 참조한다. 여러 상품이 같은 규정을 공유·재사용한다.

### 결정 3. 부분환불(취소수수료 차감)을 도입한다

예매형 시간차 환불(50%/20%)은 본질적으로 **부분환불**이다. 앞선 "전액만" 단순화를 폐기하고 부분환불을 범위에 포함한다. 환불액 = 결제액 × 환불비율, 차액은 취소수수료로 간주. 환불 이력을 누적 기록한다.

### 결정 4. 취소표 쿨다운 복원 (예매형, 되팔이 방지)

취소·환불된 재고를 **즉시 `available`로 돌리지 않고 쿨다운 후 재오픈**한다.

- 구현 후보: **① Redis 지연 큐(ZSET, score=releaseAt)** 폴러가 도래분 복원 — 스택 정합 / **③ 배치 윈도우**(정해진 시각 일괄 오픈) — 단순.
- ⚠️ **ADR-004 영향**: 재고에 4번째 상태(cooldown)가 추가된다. 불변식이 `total = available + reserved + sold + cooldown`으로 확장된다. (예매형에만 활성, 배송형은 보통 비활성)

### 결정 5. 보상환불은 하이브리드로 처리한다

- **시스템 정합성 보상**(결제 성공 후 후속 처리 실패, 중복결제 등 명백): **자동 환불(Saga)** — CS를 기다리지 않고 즉시 정합성 회복. (Kafka 이벤트, 멱등)
- **모호·분쟁성 보상**: **고객센터(CS) 수동** 환불(자동 보상 경로를 재사용해 실행).

### 결정 6. 환불 상태 머신 + 멱등성

`Payment`: `PAID → REFUND_REQUESTED → REFUNDED`(또는 `REFUND_FAILED`). 부분환불은 환불액 누적(환불 이력). 외부 PG 환불 호출은 **멱등키**로 중복 환불을 방어한다. 환불 성공 시 재고 처리(쿨다운/복원)와 정산 역처리를 동반한다.

### 결정 7. 결제 재시도(`P-P4`)를 3종으로 구분한다

| 종류 | 트리거 | 처리 |
| :--- | :--- | :--- |
| **① 시스템 자동 재시도** | PG 호출 **일시 오류**(timeout/5xx) | 지수 백오프 + 멱등키 + 최대 N회. 한도 초과 시 FAILED |
| **② 사용자 재시도** | 결제 **거절**(한도/인증 실패 등) | 예약 TTL(`P-O2`) 내에서 사용자가 다시 시도(같은 주문 유지) |
| **③ (재시도 아님) TTL 만료** | 결제 제한시간 초과 | EXPIRED → 예약 해제 → 재큐잉 |

> "일시 장애는 자동 재시도, 결정적 거절은 자동 재시도 안 함"이 핵심. 멱등키(`P-P2`)가 재시도의 안전장치다. 후방 컨슈머 재처리(오프셋/DLQ)는 ADR-002 영역으로 분리.

## 3. Consequences (결과 및 트레이드오프)

### ✅ 이점
1. **되팔이 억제**: 쿨다운으로 취소표 즉시 선점(봇)을 완화.
2. **유형 적합성**: 예매·배송 각각의 현실 규칙을 정확히 반영.
3. **재사용성**: 템플릿으로 동일 규정을 여러 상품이 공유.
4. **정합성 자동 회복**: 시스템 보상은 Saga로 자동 처리되어 데이터가 깨진 채 방치되지 않음.
5. **법적 보수성**: 자동확정 최소 7일로 청약철회권 침범 회피.

### ⚠️ 트레이드오프
1. **복잡도 증가**: 부분환불 금액 계산, 쿨다운 상태(ADR-004 4번째 버킷), 상태 머신·멱등.
2. **구현 후순위**: 정의만 확정, 구현은 핵심(Phase 2~3) 이후. 그 전까지 정책 문서가 진실원.
3. **PG 의존**: 실제 환불·부분취소는 PG 계약에 의존(데모는 모의 PG로 대체).
4. **데모 범위 한계**: 배송 이행 생명주기는 정책만 정의하고 구현은 미룸(기능 명세 §10.6).
