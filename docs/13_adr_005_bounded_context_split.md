# ADR-005. 상품 유형 분리 및 바운디드 컨텍스트 정렬 (예매형 / 배송형)

> **ADR(Architecture Decision Record)**: 상품을 **예매형(TICKET)**과 **배송형(SHIPPING)**으로 분리하고, "선착순 코어는 공유 / 이행(fulfillment)은 분기"로 바운디드 컨텍스트를 정렬하는 결정을 기록한다. 모듈러 모놀리식으로 시작해 MSA로 진화 가능한 경계를 지금 확정한다.

- **상태(Status)**: Accepted
- **일자**: 2026-06-24
- **관련 Phase**: 설계 결정 (⚠️ **구현은 후순위** — 핵심 큐·동시성(Phase 2~3) 이후)
- **결정 대상**: 상품 유형 도입, 컨텍스트 경계, 모듈 구조 진화 경로
- **관계**: ADR-001(엔티티) **refine** / 환불·이행 정책은 [`14_adr_006_refund_cancellation_policy.md`](./14_adr_006_refund_cancellation_policy.md)에서 다룸

---

## 1. Context (상황)

원래 설계(ADR-001·기획)는 모든 상품을 **단일 "한정 상품(Goods + Stock)" 모델**로 다뤘다. 이는 실수가 아니라, 이 프로젝트의 본령인 **선착순 트래픽·동시성 제어에 집중하기 위한 의도적 단순화**였다. 큐 진입·재고 예약·주문·결제까지는 티켓이든 굿즈든 동일하기 때문이다.

그러나 **환불·이행 요구사항([`14_adr_006`](./14_adr_006_refund_cancellation_policy.md))이 들어오면서 결제 *이후* 생명주기가 완전히 갈라졌다.**

| 구분 | 예매형(TICKET) | 배송형(SHIPPING) |
| :--- | :--- | :--- |
| 결제 후 흐름 | 입장권 유효화 → 관람(도래일) | 배송준비→배송중→배송완료→구매확정 |
| 환불 기준 | **도래일까지 남은 기간**(시간차 부분환불) | **구매확정** 전/후 |
| 취소 재고 | **쿨다운 후 재오픈**(취소표, 되팔이 방지) | 보통 즉시 또는 불필요 |
| 고유 속성 | `eventDate`(도래일) | `autoConfirmDays`(자동확정일), 배송상태 |

> 결제 이후 두 유형은 **언어·규칙·불변식이 다른 별개의 도메인**이다. 하나의 모델에 욱여넣으면 nullable 필드와 type 분기가 난무하는 **god model**이 된다.

---

## 2. Decision (결정)

### 결정 1. 상품 유형(`Goods.type`)을 도입한다

`Goods`에 유형 식별자를 둔다: `TICKET`(예매형) · `SHIPPING`(배송형). 유형은 결제 이후 **이행·환불 전략을 선택하는 분기 키**다.

### 결정 2. "선착순 코어는 공유, 이행은 분기"로 경계를 긋는다

통째로 둘로 복제하지 않는다. **심장(큐·재고·동시성)을 두 번 만들지 않는다.**

```
        ┌───────────── 공유 코어 (선착순 본령, 단일) ─────────────┐
요청 ─▶ │  대기열(Queue) · 재고예약(Stock) · 주문(Order) · 결제(Payment)  │
        └───────────────────────────┬───────────────────────────┘
                          결제 확정(PAID) 후 분기 (이행 단계에서 갈림)
                  ┌────────────────┴────────────────┐
          ┌───────▼────────┐                ┌────────▼────────┐
          │  예매 컨텍스트    │                │  배송 컨텍스트     │
          │ Ticketing        │                │ Commerce/Shipping │
          │ 도래일·시간차환불 │                │ 배송·구매확정·반품 │
          │ ·취소표 쿨다운    │                │ ·자동확정         │
          └────────────────┘                └─────────────────┘
```

- **공유(front)**: 대기열·재고 예약·주문·결제 — 유형 무관 동일.
- **분기(back)**: 결제 확정 이후의 이행과 환불 — 유형별 컨텍스트.

### 결정 3. 엔티티: 공통 Goods + 유형별 상세 분리

ADR-001의 `Goods`를 **공통 베이스 + 유형별 상세 테이블**로 확장한다(ID 참조, ADR-001 결정 3 정합).

| 테이블 | 보유 |
| :--- | :--- |
| `Goods`(공통) | name, description, price, openAt, **type**, maxPurchasePerMember, paymentTimeoutMinutes |
| `GoodsTicketDetail`(예매) | goodsId, **eventDate**(도래일), refundPolicyTemplateId, 취소표 쿨다운 설정 |
| `GoodsShippingDetail`(배송) | goodsId, **autoConfirmDays**(최소 7, 기본 7) |

### 결정 4. 결제 후 이행을 유형별 전략(Fulfillment)으로 추상화한다

주문 확정(PAID) 이후의 처리를 `Fulfillment` 추상으로 두고 유형별 구현(`TicketFulfillment` / `ShippingFulfillment`)을 둔다. 후방 Kafka 이벤트(ADR-002)로 느슨하게 연결.

### 결정 5. 모듈러 모놀리식으로 시작, MSA로 추출 가능한 경계를 지금 확정한다

- **현 모듈(core/api/worker)은 "기술 계층" 분리**(아키텍처 §2.2)이고, 본 결정은 **"도메인(바운디드 컨텍스트)" 분리**(세로 절단)다. **둘은 공존**한다.
- 지금은 **물리적 서비스로 쪼개지 않는다**(시기상조). `module-core` 안에서 도메인 패키지를 컨텍스트 경계(`shared` / `ticketing` / `commerce`)에 맞춰 배치하고, 모듈 간 결합을 **이벤트·인터페이스로 느슨하게** 유지한다.
- 향후 필요 시 `module-ticketing` / `module-commerce`로 **추출(Strangler Fig)**하는 비용을 최소화한다.

## 3. Consequences (결과 및 트레이드오프)

### ✅ 이점
1. **god model 회피**: 유형별 규칙·불변식이 자기 컨텍스트에 격리된다.
2. **MSA 추출 저비용**: 경계를 지금 도메인에 맞춰 그어두면, 나중 서비스 분리 시 마찰이 작다.
3. **본령 보존**: 선착순 코어가 단일로 유지되어 큐·동시성 자산이 중복·희석되지 않는다.
4. **확장성**: 신규 유형(예: 디지털 상품)도 같은 패턴(공통 + 상세 + Fulfillment)으로 추가.

### ⚠️ 트레이드오프
1. **분기 복잡도**: 유형 판별·전략 디스패치가 생긴다. 공유/분기 경계를 규율 있게 지켜야 한다.
2. **구현 후순위**: 본 결정은 **정의만 확정**하고 구현은 핵심(Phase 2~3) 이후로 미룬다. 그 사이 정책 문서가 단일 진실원이다.
3. **엔티티 증가**: 상세 테이블·조인이 늘어난다(ADR-001 §3의 의도된 비용과 동일 선상).
4. **경계 오판 위험**: 잘못 그으면 공유 코어로 분기 로직이 새어들 수 있다 → 코드리뷰·ADR로 방어.

| :--- | :--- |
| 상품 유형 도입 | `Goods.type` = TICKET / SHIPPING, 이행·환불 분기 키 |
| 코어 공유·이행 분기 | 큐·재고·주문·결제는 단일 공유, 결제 후 이행만 컨텍스트 분리 |

| Fulfillment 추상 | 결제 후 처리를 유형별 전략으로, Kafka 이벤트 연결 |
| 모듈러 모놀리식→MSA | 경계는 지금 도메인 정렬, 물리적 추출은 나중(Strangler) |
