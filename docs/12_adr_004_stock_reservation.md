# ADR-004. 재고 예약 모델 — 예약 기반 3-카운터(available / reserved / sold)

> **ADR(Architecture Decision Record)**: 한정 재고의 차감 방식을 "단순 즉시 차감"에서 **시한부 예약(soft hold) 기반 3-카운터 모델**로 확정한 결정과 근거를 기록한다.
> 본 결정은 [`6_adr_001_entity_design.md`](./6_adr_001_entity_design.md) 결정 1(Goods↔Stock 분리)의 **Stock 스키마를 확장(refine)**하며, 도메인 정책 [`11_domain_policy.md`](./11_domain_policy.md) `P-S2` 확정을 형식화한다.

- **상태(Status)**: Accepted
- **일자**: 2026-06-24
- **관련 Phase**: Phase 2(동시성 제어) · Phase 3(부하 검증)
- **결정 대상**: Stock 재고 차감 모델 및 카운터 스키마
- **관계**: ADR-001 결정 1을 **refine**(대체 아님, Stock 컬럼 확장) / `P-S2`·`P-S4`·NFR `S5`/`S6` 형식화

---

## 1. Context (상황)

한정 재고(예: 굿즈 1,000개)를 두고 수만 명이 동시에 주문할 때, 재고 차감 시점에는 **상충하는 두 단순안**이 존재한다.

| 단순안 | 동작 | 치명적 문제 |
| :--- | :--- | :--- |
| **결제 후 차감** | 결제가 성공해야 재고를 깎음 | 결제 진행 중 재고가 안 깎여 있어, 동시 통과자가 한도를 초과 → **oversell** |
| **순수 즉시 차감** | 주문 생성 시 영구 소비 | 결제를 안 하거나 실패한 사용자의 재고가 **영구히 묶임**(재고 누수) |

> 즉, "정합성(oversell 차단)"과 "재고 회수(묶임 방지)"를 **하나의 단순안으로는 동시에 만족할 수 없다.**

또한 ADR-001은 Stock을 `{ totalQuantity, remainQuantity }` 2개 카운터로 정의했으나, 이는 "예약 중 vs 판매 확정"을 구분하지 못해 위 회수 흐름과 운영 관측을 표현하기에 부족하다.

---

## 2. Decision (결정)

### 결정 1. 예약 기반(soft hold) 차감 모델을 채택한다

차감을 **영구 소비가 아니라 "시한부 예약"**으로 다룬다. 주문 시점에 재고를 **즉시·원자적으로 예약 상태로 이동**시키고, 결제 결과에 따라 **확정**하거나 **해제(복원)**한다.

- **oversell 차단**: 예약은 결제를 기다리지 않고 **주문 시점에** 가용 재고를 깎으므로, 동시 통과자가 한도를 넘을 수 없다.
- **재고 묶임 방지**: 예약에는 **만료시간(`Payment.timeoutAt`, ADR-001 결정 2)**이 붙고, 만료 시 워커가 자동 해제한다([`10_functional_spec.md`](./10_functional_spec.md) UC-07). 묶임은 결제 제한시간 동안만 유지된다(= 공정성: 먼저 잡은 자에게 결제 기회).

### 결정 2. Stock을 3개 가변 카운터 + 1개 불변 앵커로 확장한다

ADR-001의 `remainQuantity`를 `availableQuantity`로 바꾸고 `reservedQuantity`·`soldQuantity`를 추가한다.

| 컬럼 | 성격 | 의미 |
| :--- | :--- | :--- |
| `totalQuantity` | **불변 앵커** | 원본 총량. 주문 생애주기에서 불변(관리자 재입고 시에만 변경) |
| `availableQuantity` | 가변 | 아직 아무도 잡지 않은 가용 수량 |
| `reservedQuantity` | 가변 | 예약했으나 결제 전(보류) 수량 |
| `soldQuantity` | 가변 | 결제 확정된 판매 수량 |

### 결정 3. 합계 불변식을 정합성 자가검증 기준으로 둔다

```
totalQuantity == availableQuantity + reservedQuantity + soldQuantity   (항상 성립)
availableQuantity >= 0                                                  (oversell 금지)
```

`totalQuantity`는 나머지로 **유도 가능(redundant)**하지만, **합계 대조를 통한 누수 탐지**와 **원본 수량 보존**을 위해 의도적으로 별도 저장한다(denormalization).

### 결정 4. 상태 전이는 두 카운터의 원자적 이동으로 정의한다

| 이벤트 | 카운터 이동 | 가드 |
| :--- | :--- | :--- |
| **주문 생성** | `available-- , reserved++` | `available > 0` (없으면 품절) |
| **결제 성공** | `reserved-- , sold++` | — |
| **만료/실패/취소** | `reserved-- , available++` | — |

```
                  주문생성                  결제성공
 available ───────────────▶ reserved ───────────────▶ sold
     ▲                          │
     └──────────────────────────┘
              만료/실패 (해제)
```

전이마다 **두 카운터를 한 트랜잭션·한 임계구역에서 함께** 갱신한다. 동시성 보호는 단일 DB = **JPA 비관적 락(`FOR UPDATE`, Stock 단일 행)**, Scale-out = **Redis 분산 락**(아키텍처 §2.4 전략 3, `P-S3`).

## 3. Consequences (결과 및 트레이드오프)

### ✅ 이점
1. **두 문제 동시 해결**: 주문 시점 `available > 0` 가드로 oversell 0건(NFR `S5`)을 보장하면서, TTL 해제로 재고 묶임을 방지한다.
2. **정합성 자가검증**: 합계 불변식이 깨지면 동시성 버그(카운터 누락)를 즉시 탐지하는 안전망이 된다.
3. **운영 관측성**: "현재 예약 중 N개 / 판매 M개 / 매진율 sold/total"을 Stock 한 행에서 즉시 조회. 별도 Order 집계 불필요.
4. **회복 경로 명확화**: 결제 타임아웃 → 해제 → 다음 대기자 입장(`P-O2`/`P-Q4`)이 카운터 이동으로 일관되게 표현된다.

### ⚠️ 트레이드오프
1. **전이 동기화 부담**: 모든 전이가 **두 카운터를 원자적으로** 옮겨야 한다. 한쪽만 갱신하는 버그는 불변식 위반으로 이어진다 → 동시성 테스트로 합계 불변식을 반드시 검증한다.
2. **의도된 중복(denormalization)**: `totalQuantity`는 유도 가능하지만 검증·의미 보존을 위해 중복 저장한다. 재입고 시 total과 available을 함께 조정해야 한다.
3. **단순안 대비 복잡도 증가**: 단일 카운터(A안)보다 상태가 많다. 그러나 본 프로젝트의 정체성(재고·동시성 통제)에 비추어 **관측성·검증 이득이 비용을 상회**한다고 판단한다.
4. **멱등성 필요**: 결제 콜백/만료 정리가 중복 실행되어도 카운터가 이중 이동하지 않도록 멱등 처리(`P-P2`)와 함께 구현한다.

## 4. 결정 요약

| 결정 | 한 줄 요약 |
| :--- | :--- |
| 예약 기반 차감 | 주문=예약(soft hold), 결제=확정, 만료=해제 → oversell·묶임 동시 해결 |
| 3-카운터 + 앵커 | Stock = `total`(불변) + `available`·`reserved`·`sold`(가변) |
| 합계 불변식 | `total = available+reserved+sold` 로 누수 자가검증, `available>=0` |
| 원자적 전이 | 전이마다 두 카운터를 한 임계구역(비관적 락/분산 락)에서 함께 이동 |
