# ADR-002. 큐 아키텍처 — Redis ZSET(전방 대기열) + Kafka(후방 이벤트 파이프라인) 하이브리드

> **ADR(Architecture Decision Record)**: 대기열 및 비동기 처리에 사용할 큐 시스템 선정 결정과 근거를 기록한다.

- **상태(Status)**: Accepted
- **일자**: 2026-06-19
- **관련 Phase**: Phase 3(Redis 대기열) · Phase 4(Kafka 이벤트 파이프라인)
- **결정 대상**: "선착순 진입 대기열"과 "주문/결제 비동기 처리·MSA 통신"에 각각 사용할 큐 기술

---

## 1. Context (상황)

**Schale Queue**의 요청 흐름은 성격이 전혀 다른 두 구간으로 나뉜다.

```
[수만 명 동시 진입]
      │
      ▼
┌───────────────────────┐
│ ① 전방: 선착순 진입 대기열 │  ← 실시간 순번 조회, 초고속, UX 직결
└───────────┬───────────┘
            │  (입장 토큰 획득)
            ▼
┌───────────────────────┐
│ ② 후방: 주문/결제 처리    │  ← 데이터 무결성, 유실 불가, MSA 통신
└───────────────────────┘
```

- **① 전방 대기열**: 폭증 트래픽을 평탄화하고, 사용자에게 **"당신은 지금 12,345번째"** 같은 실시간 순번을 보여줘야 한다. 연산은 단순하지만 **초당 수만 건의 순위 조회**가 발생한다. 데이터가 일부 휘발되어도(=대기 도중 이탈) 치명적이지 않다.
- **② 후방 처리**: 대기열을 통과한 요청의 주문·결제는 **단 한 건도 유실되어서는 안 된다.** 또한 주문/결제/알림/정산 등 여러 서비스가 느슨하게 협력(MSA)해야 하므로, 시스템 간 결합도를 낮추는 메시징이 필요하다.

> **하나의 큐 기술로 두 구간을 모두 감당하려 하면 어느 한쪽이 반드시 희생된다.** 따라서 구간별 특성에 맞는 **하이브리드** 전략이 필요하다.

---

## 2. Decision (결정)

| 구간 | 채택 기술 | 핵심 역할 |
| :--- | :--- | :--- |
| **① 전방 진입 대기열** | **Redis Sorted Set (ZSET)** | 선착순 순번 관리 및 실시간 순위 조회(`ZRANK`/`ZADD`) |
| **② 후방 이벤트 파이프라인** | **Apache Kafka** | 주문/결제 이벤트의 영속적·비동기 전파 및 MSA 간 통신 |

- **전방**: 요청을 Redis ZSET에 `ZADD`(score = 진입 timestamp)로 적재하고, `ZRANK`로 실시간 대기 순번을 조회한다. 워커가 정해진 처리량만큼 앞에서부터 입장시킨다.
- **후방**: 입장·주문·결제 이벤트를 Kafka 토픽으로 발행(Pub)하고, 각 컨슈머(주문/결제/알림/정산 서비스)가 구독(Sub)하여 **이벤트 기반(Event-Driven)**으로 처리한다.

---

## 3. Consequences (기술적 근거 및 트레이드오프)

### 3.1. 왜 전방은 Redis ZSET인가

| 강점 | 설명 |
| :--- | :--- |
| **초고속 메모리 연산** | 모든 데이터가 메모리에 상주. `ZADD`/`ZRANK`가 O(log N)으로, 수만 동시 요청의 순번 조회를 ms 단위로 처리 |
| **실시간 순번 조회(UX)** | `ZRANK`로 "내 앞에 몇 명 남았는지"를 즉시 반환 → 대기 화면 UX에 직결 |
| **정렬·범위 연산 내장** | score(timestamp) 기반 자동 정렬, `ZRANGE`로 상위 N명 일괄 입장 처리 용이 |

> **트레이드오프**: 메모리 기반이라 영속성이 약하다(장애 시 대기열 유실 가능). 그러나 대기열 데이터는 **휘발되어도 재진입으로 복구 가능한 성격**이므로 수용 가능한 리스크다. (AOF로 일부 보강)

### 3.2. 왜 후방은 Kafka인가

| 강점 | 설명 |
| :--- | :--- |
| **디스크 기반 영속성** | 메시지를 디스크 로그에 보존 → **주문/결제 이벤트 무유실 보장**. 컨슈머 장애 후 재시작해도 오프셋부터 재처리 |
| **Pub/Sub & 낮은 결합도** | 발행자는 컨슈머를 알 필요 없음 → 주문/결제/알림/정산 서비스를 **독립적으로 확장·배포**(MSA) |
| **재처리·순서 보장** | 파티션 단위 순서 보장 + 오프셋 재처리로, 결제 실패/재시도 같은 복잡한 흐름에 견고 |

> **트레이드오프**: Redis처럼 "N번째 메시지"를 실시간 역추적(랜덤 순위 조회)하는 데는 부적합하고, 운영 복잡도(브로커/주키퍼-혹은-KRaft, 파티션 설계)가 높다. 그러나 후방 구간이 요구하는 것은 **순위 조회가 아니라 무결성**이므로 이 단점은 무관하다.

### 3.3. 하이브리드의 종합 이점

- **책임 분리**: "실시간성(Redis)"과 "무결성(Kafka)"을 각 구간에 최적 배치 → 어느 쪽도 타협하지 않음.
- **장애 격리**: 전방 대기열 장애가 후방 결제 데이터를 오염시키지 않고, 후방 처리 지연이 전방 UX를 막지 않음.
- **확장 경로**: 후방을 Kafka 이벤트로 풀어두면 신규 컨슈머(예: 데이터 분석, 재고 동기화) 추가가 기존 코드 변경 없이 가능.

### 3.4. 향후 관리 포인트

1. **Redis 영속성 한계**: 대기열 유실 시 사용자 재진입 정책(순번 보존 여부)을 명확히 정의해야 한다.
2. **Kafka 운영 비용**: 로컬/단일 노드로 시작하되, 토픽·파티션·컨슈머 그룹 설계와 모니터링 체계를 Phase 4에서 함께 수립한다.
3. **이벤트 스키마 관리**: 이벤트 페이로드의 하위 호환성(스키마 진화) 전략을 별도 ADR로 다룰 수 있다.
4. **정확히 한 번(exactly-once)**: 결제 같은 민감 이벤트는 멱등 처리(idempotency key) 또는 트랜잭셔널 아웃박스 패턴을 병행 검토한다.

---

## 4. 결정 요약

| 결정 | 한 줄 요약 |
| :--- | :--- |
| 전방 = Redis ZSET | **실시간 순번 조회**가 생명인 진입 대기열 → 메모리 초고속 + `ZRANK` |
| 후방 = Kafka | **무유실·낮은 결합도**가 생명인 주문/결제 → 디스크 영속 + Pub/Sub |
| 하이브리드 | 실시간성과 무결성을 **구간별로 분리 최적화** |
