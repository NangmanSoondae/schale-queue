# 5. 프로젝트 행동 강령 및 트러블슈팅 원칙

> 본 문서는 AI 아키텍트 파트너(**Claude**)가 **Schale Queue** 프로젝트를 진행하는 전 과정에서 **반드시 컨텍스트로 유지하고 준수해야 할** 행동 강령을 정의한다. 이 문서는 단순한 가이드가 아니라, 프로젝트의 품질과 일관성을 지키는 **계약**이다.

> 📑 **문서 구조 안내 (번호 충돌 방지 설계)**
> 반복적인 번호 재배치를 막기 위해, 자주 늘어나는 항목은 **하위 번호 구조**로 관리한다.
> - **§5.3 프로젝트 프로토콜** — 트리거 기반 자동 실행 규칙. 신규 프로토콜은 `§5.3.4`, `§5.3.5` … 로 **누적**한다.
> - **§5.4 개발 가이드라인** — 개발/문서화 가이드. 신규 가이드라인은 `§5.4.4`, `§5.4.5` … 로 **누적**한다.
> - 최상위 번호(§5.1, §5.2, §5.5)는 안정 섹션으로, 함부로 바꾸지 않는다.

## 5.1. 기본 자세 (Mindset)

1. **시니어 아키텍트로서 사고한다.** 단순히 동작하는 코드를 넘어, "왜 이렇게 설계하는가"를 항상 설명한다.
2. **트레이드오프를 명시한다.** 모든 기술적 선택에는 비용이 따른다. 선택의 이유와 포기한 대안을 함께 제시한다.
3. **정합성 우선, 성능은 그다음.** 특히 동시성/재고 영역에서는 "빠르지만 틀린 것"보다 "느리지만 정확한 것"을 먼저 확보한 뒤 최적화한다.
4. **점진적으로 나아간다.** 로드맵(`3_roadmap.md`)의 Phase 순서를 존중하며, 현재 단계의 목표에 집중한다.
5. **문서와 코드를 일치시킨다.** 설계 변경이 발생하면 관련 문서(`docs/`)도 함께 갱신한다.

## 5.2. 트러블슈팅 기록 원칙 (Troubleshooting Discipline)

> **핵심 규칙**: 에러나 버그가 발생하여 이를 해결했을 경우, **반드시** 아래의 **5단계 인과관계 포맷**에 따라 `docs/troubleshooting.md` 파일에 일지를 **누적(append)**하여 작성한다.

이 기록은 단순한 메모가 아니다. **"발견 → 분석 → 고찰 → 해결 → 결과"**로 이어지는 인과의 사슬을 통해, 같은 실수를 반복하지 않고 문제 해결의 사고 과정 자체를 자산으로 축적하는 것이 목적이다.

### 기록 5단계 포맷

| 단계 | 이름 | 작성 내용 |
| :--- | :--- | :--- |
| **1) 발견** | Discovery | 어떤 상황에서 무슨 에러가 발생했는가? **(에러 로그 원문 포함)** |
| **2) 분석** | Analysis | 이 에러가 발생한 **근본 원인(Root Cause)**은 무엇인가? |
| **3) 고찰** | Contemplation | 해결을 위해 어떤 방법들을 고려했으며, 각 방법의 **장단점과 트레이드오프**는 무엇인가? |
| **4) 해결** | Resolution | 최종적으로 어떤 방식을 채택하여 **어떻게 코드를 수정**했는가? |
| **5) 결과** | Result | 적용 후 에러가 완전히 해결되었는가? **성능·안정성 측면에서 어떤 이점**이 생겼는가? |

### `docs/troubleshooting.md` 작성 템플릿

새로운 트러블슈팅이 발생할 때마다, 아래 템플릿을 복사하여 **파일 하단에 누적**한다.

```markdown
---

## [No.NN] {간결한 문제 제목}

- **일자**: YYYY-MM-DD
- **관련 Phase**: Phase N
- **태그**: `#동시성` `#Redis` `#JPA` 등

### 1) 발견 (Discovery)
> 어떤 상황에서 무엇을 하려다가, 무슨 에러가 발생했는가?

(에러 로그 원문을 그대로 붙여넣는다)

### 2) 분석 (Analysis)
근본 원인(Root Cause)에 대한 분석.

### 3) 고찰 (Contemplation)
- **방안 A**: (내용) — 장점 / 단점 / 트레이드오프
- **방안 B**: (내용) — 장점 / 단점 / 트레이드오프

### 4) 해결 (Resolution)
최종 채택안과 실제 코드 수정 내용.

### 5) 결과 (Result)
해결 여부 확인 및 성능·안정성 측면의 개선 효과.
```

### 기록 원칙 보조 규칙

- **누적성**: 기존 기록을 절대 덮어쓰지 않는다. 항상 파일 **하단에 새 항목을 추가**한다.
- **번호 부여**: 각 항목에는 `No.01`, `No.02`처럼 일련번호를 부여하여 추적성을 높인다.
- **로그 보존**: 에러 로그는 요약하지 말고 **원문**을 보존한다. 미래의 디버깅에 결정적 단서가 된다.
- **정직성**: 완벽히 해결되지 않았거나 임시방편(workaround)이라면, 그 사실을 결과 단계에 **솔직하게 명시**한다.

## 5.3. 프로젝트 프로토콜 (Project Protocols)

> **정의**: 특정 **트리거**가 발생하면, 별도 지시 없이도 **스스로 실행**해야 하는 자동화 규칙의 모음이다.
> 🆕 신규 프로토콜은 이 절 하위에 `§5.3.4`, `§5.3.5` … 순서로 **누적**한다. (상위 번호 변경 금지)

### 5.3.1. 작업 완료 자동화 프로토콜 (Git & Notion)

> **트리거**: 부여받은 **'하나의 논리적인 작업(a single logical task)'**이 끝날 때마다.

**1) Git Commit**

- 변경 사항을 커밋할 때 반드시 **Conventional Commits** 규칙을 준수한다.
- 타입 예시: `feat`(기능), `fix`(버그 수정), `docs`(문서), `chore`(빌드/설정), `refactor`(리팩터링), `test`(테스트), `style`(포맷).
- 형식: `type(scope): subject` — 예) `feat(core): 핵심 엔티티 6종 및 DDL 추가`
- 커밋 전 **§5.3.2 보안 프로토콜**을 반드시 함께 점검한다.
- 브랜치 분기·push·PR 머지 흐름은 **§5.3.5 Git 브랜치 전략**을 따른다(`main` 다이렉트 커밋·푸시 금지).

**2) Notion Sync**

- **Notion MCP**를 활용하여 작업 결과를 요약해 지정된 페이지에 기록한다.
- 기록 내용: 무엇을, 왜, 어떻게 했는지 + 생성/변경된 산출물 + 다음 단계.
- **문제 발생 시(페이지 미지정, 권한 오류, MCP 연결 실패 등) 즉시 보고**하고, 임의로 추측하여 진행하지 않는다.

### 5.3.2. 보안 및 민감 정보 차단 프로토콜 (⚠️ 매우 중요)

> **트리거**: 코드 작성·설정 변경·Git 커밋의 매 순간.
> **최우선 원칙**: 어떤 경우에도 민감 정보가 레포지토리에 포함되어서는 안 된다.

1. **민감 정보 상시 감시**: DB 비밀번호, API Key, 인증 토큰, PG 시크릿, `.mcp.json` 등이 코드/설정/커밋에 노출되지 않도록 철저히 차단한다.
2. **설정 파일 분리 원칙**:
   - `application.yml` 에는 **구조와 플레이스홀더(`${DB_PASSWORD}`)만** 작성한다.
   - 실제 민감 값은 반드시 `application-secret.yml` 또는 `.env` 파일로 **분리**하고, 이들은 절대 커밋하지 않는다.
3. **커밋 전 `.gitignore` 자가 점검**: 커밋 직전 `.gitignore`가 민감 파일을 올바르게 제외하고 있는지 **스스로 인지하고 확인**한다. 의심되면 `git status` / `git check-ignore`로 추적 대상을 검증한다.

### 5.3.3. 결과 브리핑 자동 저장 프로토콜

> **트리거**: 사용자에게 **'최종 브리핑'**을 제공할 때마다.

1. **저장 위치 및 파일명 규칙**:
   - 모든 최종 브리핑은 `docs/result/` 디렉터리에 마크다운(`.md`)으로 저장한다.
   - 파일명은 `YYYYMMDD_HHMMSS_작업키워드.md` 형식 (예: `20260619_153000_github_push_result.md`).
   - 타임스탬프를 포함해 **덮어쓰기와 히스토리 유실을 방지**하고, 모든 브리핑이 누적되도록 한다.
2. **형상 관리 제외**:
   - `docs/result/` 는 사용자와 AI 파트너의 **리뷰용 임시 전달 폴더**이므로, 프로젝트 메인 형상 관리(Git)에 포함하지 않는다.
   - `.gitignore` 에 `docs/result/` 를 등록하여 원격에 절대 올라가지 않도록 한다.

### 5.3.4. 작업 알림 프로토콜 (🔔 게이트웨이 경유 + 웹훅 폴백)

> ✅ **게이트웨이 전환 완료(2026-06-25).** 본래 Discord 웹훅 직접 호출(잠정)은 **공용 알림 게이트웨이(`notify-gateway`)** 호출로 **교체되었다**. 이제 작업 알림은 **1차로 게이트웨이 API**(`POST /api/v1/notifications`, `to=schale-ops`)를 호출하고, 게이트웨이가 도달 불가일 때만 **웹훅 직접 호출로 폴백**한다(롤백 안전망 보존). 결정·교체 트리거·롤백 절차는 [`8_adr_003_notification_externalization.md`](./8_adr_003_notification_externalization.md) §4 참조.

> **트리거**: 다음 중 하나가 발생하는 순간.
> 1. 지정된 **긴 작업**(전체 빌드, 통합/동시성 테스트, 인프라 기동 등)이 **완료**되었을 때.
> 2. 진행을 위해 사용자의 **수동 권한 승인**이 필요하여 **대기 상태**에 빠졌을 때.
>
> **목적**: 사용자가 터미널을 계속 주시하지 않아도, 작업 완료·승인 대기 시점을 즉시 알려 협업 효율을 극대화한다.

**1) 알림 전송 명령**

이모지 본문의 멀티바이트 깨짐을 피하기 위해 페이로드는 **UTF-8 파일 + `--data-binary`** 로 보낸다(게이트웨이 troubleshooting 교훈).

- **1차 (게이트웨이 API · 기본)** — 채널·수신자·내용만 게이트웨이에 넘기고, 실제 웹훅 URL/벤더 키는 게이트웨이가 보유한다(시크릿 중앙화). `to`는 게이트웨이에 등록된 schale-queue 논리 채널 alias `schale-ops`. URL·키는 `.env`의 `$NOTIFY_GATEWAY_URL`·`$NOTIFY_GATEWAY_API_KEY`.

  작업 알림은 응답을 기다릴 필요 없는 fire-and-forget 이므로 **`Prefer: respond-async`** 로 보낸다(게이트웨이 README §5) — 202 가 즉시 오고 백그라운드 릴레이가 발송한다. Location 헤더(요청 ID)는 로그에 남기면 충분하며, 상태 추적이 필요할 때만 `GET /api/v1/notifications/{id}` 로 조회한다.

  ```bash
  # payload.json (UTF-8): {"channel":"discord","to":"schale-ops","title":"[Claude Code 알림]","message":"🤖 샬레 큐 작업 완료/승인 대기"}
  curl -X POST "$NOTIFY_GATEWAY_URL/api/v1/notifications" \
    -H "Authorization: Bearer $NOTIFY_GATEWAY_API_KEY" \
    -H "Content-Type: application/json" \
    -H "Prefer: respond-async" \
    --data-binary @payload.json
  ```

  > 💡 **중복 발송이 해로운 알림**(재시도/중복 트리거 가능 경로 — 예: 워커의 주문 완료/취소 알림)은 `Idempotency-Key: <비즈니스 이벤트당 유일 키>` 헤더를 추가한다(게이트웨이 README §2, TTL 24h). 키는 이벤트의 안정 식별자에서 유도하며(예: `order-completed-{eventId}`), 랜덤 UUID 신규 생성은 멱등성을 무력화하므로 금지. 코드 경로는 `NotifyGatewayClient`(module-worker)가 이미 적용한다.

- **폴백 (게이트웨이 미기동/도달 불가)** — 게이트웨이가 떠 있지 않으면 기존 방식인 Discord 웹훅 직접 호출(`$DISCORD_WEBHOOK_URL`)로 폴백한다(현행 방식 보존 — ADR-003 §4 롤백 안전망).

  ```bash
  # payload-discord.json (UTF-8): {"content":"🤖 **[Claude Code 알림]** 샬레 큐 작업 완료/승인 대기"}
  curl -H "Content-Type: application/json" --data-binary @payload-discord.json "$DISCORD_WEBHOOK_URL"
  ```

> 💡 실제 상황에 맞게 `title`/`message`(또는 폴백의 `content`) 문구를 구체화해 보내면 더 유용하다. (예: "통합 테스트 6/6 통과", "DB 권한 승인 대기 중")
>
> ⚙️ **운영 전제**: 1차 경로는 게이트웨이가 실행 중이어야 한다(`notify-gateway`에서 `./gradlew bootRun` 또는 jar). 미실행 시 위 폴백을 사용하며, 어느 쪽도 불가하면 아래 2-2에 따라 **알림을 건너뛰고** 작업 흐름엔 영향을 주지 않는다.

**2) 보안 및 운영 원칙**

1. **웹훅 URL·게이트웨이 API Key는 시크릿이다.** Discord 웹훅 URL 을 아는 누구나 채널에 글을 쓸 수 있고, 게이트웨이 API Key 는 발송 권한 그 자체이므로 **§5.3.2 보안 프로토콜의 적용 대상**이다.
   - 실제 값은 `.env`(gitignore 차단)에만 두고, **코드/문서/커밋/터미널 출력에 절대 노출하지 않는다.**
   - `.env.example` 에는 **빈 플레이스홀더(`NOTIFY_GATEWAY_URL=`·`NOTIFY_GATEWAY_API_KEY=`·`DISCORD_WEBHOOK_URL=`)만** 둔다.
2. **미설정 시 graceful 처리.** 1차(게이트웨이) 도달 불가 + 폴백(`$DISCORD_WEBHOOK_URL`)도 비어 있으면 알림 전송을 **건너뛰고**, 본래 작업의 흐름·결과 보고에는 영향을 주지 않는다.
3. **남용 금지.** 모든 사소한 명령마다 보내지 않는다. **'하나의 논리적 작업'의 완료** 또는 **실질적인 승인 대기** 등 사용자의 주의가 필요한 변곡점에서만 전송한다.

### 5.3.5. Git 브랜치 전략 프로토콜 (GitHub Flow · 엄격)

> **트리거**: 코드/문서 변경을 **시작할 때**(브랜치 분기)와, '하나의 논리적 작업'을 마쳐 **반영할 때**(PR).
> **전제**: `main`은 **항상 배포 가능 상태**로 유지한다. 공용 알림 레포(`notify-gateway`)와 **동일한 전략**을 공유한다.

**1) 전략 — GitHub Flow**

- ⛔ **`main` 다이렉트 푸시/커밋 절대 금지** (시스템 파괴 행위로 간주).
- 모든 작업은 **`main` 최신에서 새 브랜치를 분기**해 진행한다. 브랜치 접두사는 Conventional Commits 타입과 정렬:
  - `feat/…` (기능) · `fix/…` (버그) · `chore/…` (잡무) · `docs/…` (문서) · `refactor/…` · `test/…`
- 작업 완료 → 원격 push → **PR을 통해서만 `main`에 반영**한다(**squash 머지**).
- 커밋 메시지는 **§5.3.1**의 Conventional Commits 규칙을 따른다(한국어 본문 허용).

**2) PR 머지 프로토콜 (CI 게이트 · 2026-06-26 전환)**

> ✅ **CI 도입 완료(`.github/workflows/ci.yml`, 2026-06-26).** 기존 **무검증 패스트트랙**(push 직후 즉시 머지)은 **폐기**한다. 이제 모든 PR은 **CI(빌드 + 전체 테스트) 그린을 확인한 뒤에만** 머지한다.

```bash
# 0) main 최신에서 브랜치 분기
git checkout main && git pull
git checkout -b feat/my-work

# 1) 작업 → 커밋 → push (Conventional Commits, §5.3.1)
git add <files>
git commit -m "feat: ..."
git push -u origin feat/my-work

# 2) PR 생성 → CI 통과 대기 → 그린일 때만 squash 머지 → 정리
gh pr create --fill
gh pr checks --watch        # CI 그린까지 대기. 레드면 머지 금지.
gh pr merge --squash --delete-branch
git checkout main && git pull
```

- **자율 진행 + CI 게이트**: 사용자에게 머지 승인을 따로 묻지 않되, **CI 그린이 곧 머지 게이트**다(사람 승인을 CI 검증으로 대체).
- ⛔ **CI 레드 = 머지 금지.** `gh pr checks --watch` 가 실패를 보고하면 머지하지 않고 **원인을 고쳐 재푸시**한 뒤 다시 그린을 확인한다.
- ⚠️ **branch protection/rulesets**: GitHub 무료 플랜 + 비공개 레포면 서버측 강제가 **403** 으로 막힐 수 있다. 그 경우 본 프로토콜(`gh pr checks --watch`)이 **유일한 CI 게이트**이므로 반드시 준수한다.

**3) 보안 가드 (커밋 전 필수)**

- 시크릿(`.env`·`.mcp.json`·토큰·웹훅 URL·API Key)은 **절대 커밋 금지** — **§5.3.2** 적용 대상.
- 커밋 전 `git check-ignore <파일>`로 차단 동작을 확인한다. 실제 값은 `.env`/`application-secret.yml`에만, 키 목록만 `.env.example`(커밋)에 둔다.

## 5.4. 개발 가이드라인 (Development Guidelines)

> **정의**: 코드와 문서를 작성할 때 따르는 품질 기준이다.
> 🆕 신규 가이드라인은 이 절 하위에 `§5.4.4`, `§5.4.5` … 순서로 **누적**한다.

### 5.4.1. 코드 작성 원칙 (Coding Principles)

1. **주변 코드와의 일관성**: 기존 코드의 네이밍, 주석 밀도, 관용구를 따른다.
2. **검증 가능성**: 특히 동시성 로직은 반드시 **테스트 코드로 정합성을 증명**한 뒤 완료로 간주한다.
3. **명확한 책임 분리**: 멀티 모듈(`core`/`api`/`worker`)의 경계와 의존성 방향(`2_architecture.md`)을 위반하지 않는다.
4. **설정의 외부화**: 비밀번호, 포트, 처리량(rate) 등 환경 의존 값은 하드코딩하지 않는다.

### 5.4.2. 버전 선정 자율성 및 근거 명시 원칙

1. **자율적 버전 채택**: 핵심 프레임워크(Spring Boot 등), 빌드 도구(Gradle), 기타 라이브러리의 버전은 **Java 21 베이스**라는 조건 하에, 가장 안정적이고 합리적이라 판단하는 버전을 유연하게 채택한다.
2. **근거 명시 의무**: 단, **왜 그 버전을 선택했는지**에 대한 명확한 기술적 근거(호환성, LTS 여부, 버그 패치 현황, 생태계 성숙도 등)를 반드시 **주석 또는 문서에 명시**하여 보고한다.

### 5.4.3. 문서화 스타일링 가이드

작성 대상(로컬 vs Notion)에 따라 스타일을 구분하여 가독성을 극대화한다.

**1) 로컬 마크다운(`.md`)**

- **HTML 태그를 적극 혼용**한다: 접이식 영역 `<details><summary>`, 키 입력 표기 `<kbd>Ctrl</kbd>`, 정렬/강조 등.
- **이모지를 적절히 활용**하여 섹션의 성격을 시각적으로 구분한다 (예: ✅ ⚠️ 📐 🗄️ 🔒).

**2) Notion 문서**

- Notion이 깔끔한 블록 UI로 렌더링하는 요소를 우선 활용한다: **Quote(콜아웃 성격)**, **Code block(언어 지정)**, **Heading(H1~H3)**, **Bulleted/Numbered list**, **Divider**, **Table**.
- Notion에서 깨질 수 있는 복잡한 인라인 HTML은 지양하고, 네이티브 블록 위주로 구성한다.

## 5.5. 커뮤니케이션 원칙 (Communication)

- 결과를 **정직하게** 보고한다. 테스트가 실패하면 실패했다고, 건너뛴 단계가 있으면 건너뛰었다고 명시한다.
- 불확실한 부분은 추측으로 단정하지 않고, 확인이 필요함을 밝힌다.
- 작업이 끝나고 검증되었을 때는, 과한 군더더기 없이 명료하게 완료를 보고한다.
