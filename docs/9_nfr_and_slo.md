# 9. 비기능 요구사항(NFR) 및 목표 지표(SLO)

> **Schale Queue**의 정체성은 "대용량 트래픽 제어"다. 기능이 "동작하는가"를 넘어 **"얼마나 잘 버티는가"**를 정량 목표로 고정한다.
> 이 문서의 숫자는 **부하 테스트의 합격/불합격 기준**이며, Phase 3·5의 검증 리포트가 이 표를 직접 채운다.
>
> ⚠️ **상태**: 시나리오 매트릭스 **확정**(2026-06-24). 지연 임계(p99 등)는 **목표값(실측으로 보정)**이다.

---

## 9.1. 기준 시나리오 매트릭스 (Reference Scenario Matrix)

단일 시나리오가 아니라 **상품 유형(ADR-005) × 규모 등급**의 매트릭스로 측정한다. 핵심 스트레스 변수는 절대 수치가 아니라 **경합비(Contention Ratio) = 동시 진입 ÷ 재고**다.

**3축 등급 (동시 진입은 두 유형 공통)**

| 등급 | 동시 진입(공통) | 예매형(TICKET) 재고 | 배송형(SHIPPING) 재고 |
| :--- | :--- | :--- | :--- |
| 소(S) | 100 | 100석 | 10개 |
| 중(M) | 1,000 | 1,000석 | 100개 |
| 대(L) | 10,000 | 5,000석 | 1,000개 |

- **동시 진입**: `openAt` 직후 수 초 내 대기열 진입 시도.
- **측정 환경**: 단일 노드(하드웨어 스펙 기록) 기준, scale-out은 별도 표기(ADR-001 §3 정합).

**대표 벤치마크 셀** — 이 4개로 압축 측정한다.

| 셀 | 구성 | 경합비 | 검증 초점 |
| :-- | :--- | :--- | :--- |
| **B1 기준부하** | 중트래픽 × 중재고 (예매 1,000석 / 배송 100개) | 1:1 ~ 10:1 | 일반 성능 기준선(지연 SLO 충족) |
| **B2 동시성 극한** | 대트래픽 × 최소재고 (**배송 10개 × 10,000명**) | **1,000:1** | **오버셀 0건**(락 경합 최악) |
| **B3 처리량 극한** | 대트래픽 × 대재고 (**예매 5,000석 × 10,000명**) | 2:1 | 큐 평탄화·입장률·완판 속도 |
| **B4 스파이크** | 오픈 순간 전원 `t=0` 버스트 | — | 순간 폭주 견딤(전방 큐) |

> 이 매트릭스는 ADR-001(읽기/쓰기 분리)·ADR-002(전방 ZSET / 후방 Kafka)가 가정한 "선착순 오픈 순간"을 유형·규모별로 구체화한 것이다.

## 9.2. 목표 지표 (SLO) — 정량 합격 기준

| # | 지표 | 목표값 | 적용 셀(§9.1) | 측정 지점 | 검증 |
| :-- | :--- | :--- | :--- | :--- | :--- |
| S1 | **대기열 진입(enqueue) 지연** | p99 < 100ms (B1 충족, B3 열화 추적) | B1·B3·B4 | `POST /queue/enter` (ZADD) | Phase 3 |
| S2 | **순번 조회 지연** | p99 < 50ms (B1 충족, B3 추적) | B1·B3 | `GET /queue/rank` (ZRANK) | Phase 3 |
| S3 | **입장 처리율(throughput)** | DB 유입을 **평탄화**(단일노드 목표 200~500 TPS, 실측 역산) | B3 | Worker dequeue rate | Phase 3 |
| S4 | **주문 임계구역 지연** | p99 < 200ms (B1 충족, B2 추적) | B1·B2 | 주문 생성 + 재고 예약 트랜잭션 | Phase 2/3 |
| S5 | **오버셀(oversell)** | **0건** (재고 = 정확히 판매수) | **모든 셀, 특히 B2** | Stock 카운터 (sold ≤ total) | Phase 2 동시성 테스트 |
| S6 | **재고 정합성** | `available ≥ 0` **및** `total = available+reserved+sold(+cooldown)` 항상 성립 | **모든 셀, 특히 B2** | DB 불변식 (P-S1) | Phase 2 |
| S7 | **에러율** | 오픈 피크 5xx < 0.1% | B2·B3·B4 | Actuator/게이트웨이 | Phase 3 |
| S8 | **주문/결제 이벤트 무유실** | 유실 0건 (at-least-once 이상) | B2·B3 | Kafka 컨슈머 오프셋 | Phase 4 |

> **등급별 해석**: 정합성(S5/S6)은 **모든 셀에서 무조건 통과**(특히 경합비 1,000:1의 B2). 지연(S1/S2/S4)은 **기준부하 B1에서 목표 충족**하고 대규모(B3)에서는 **열화를 추적·기록**한다(상한 내 유지가 목표).
>
> **S3 보충**: 입장 처리율은 "느릴수록 안전, 빠를수록 UX 좋음"의 트레이드오프 파라미터다. 목표는 **DB가 견딜 수 있는 상한 이하로 트래픽을 깎는 것**이며, 정확한 값은 S4·S7이 깨지지 않는 최대치를 B3에서 역산한다.

## 9.3. 비기능 요구사항 (NFR) — 품질 속성별

### 9.3.1. 성능 (Performance)
- 대기열 연산(ZADD/ZRANK)은 **메모리 O(log N)** 특성을 유지한다(ADR-002 §3.1).
- 응답 경로의 무거운 작업(알림·정산)은 **사용자 응답 경로에서 분리**한다(후방 Kafka, ADR-002 §2).
- I/O 바운드 구간은 **Virtual Threads**로 처리해 적은 플랫폼 스레드로 고동시성을 확보한다(아키텍처 §2.4 전략 1).

### 9.3.2. 정합성 · 신뢰성 (Consistency / Reliability)
- **재고 초과 판매 0건**은 타협 불가한 최우선 불변식이다(S5/S6).
- 단일 DB: **JPA 비관적 락**(`SELECT ... FOR UPDATE`)으로 Stock 단일 행 직렬화(ADR-001 §2).
- Scale-out: **Redis 분산 락**으로 임계구역을 클러스터 전역 직렬화(아키텍처 §2.4 전략 3).
- 후방 결제 이벤트는 **멱등 처리(idempotency key)** 또는 **트랜잭셔널 아웃박스**로 중복/유실 방어(ADR-002 §3.4).

### 9.3.3. 가용성 · 회복력 (Availability / Resilience)
- **장애 격리**: 전방 대기열 장애가 후방 결제 데이터를 오염시키지 않는다(ADR-002 §3.3).
- **Graceful Degradation**: 대기열 장애 시 신규 입장을 차단하고 재시도를 안내하되, **이미 입장한 주문 흐름은 보호**한다.
- **결제 타임아웃 복원**: 만료된 결제는 워커가 정리하고 **재고를 복원**한다(`Payment.timeoutAt`, ADR-001 §2; 상세는 [`11_domain_policy.md`](./11_domain_policy.md)).

### 9.3.4. 확장성 (Scalability)
- ID 참조 기반 애그리거트 분리로 **도메인별 DB 분리/샤딩 여지**를 확보(ADR-001 §3).
- 후방을 Kafka 이벤트로 풀어 **신규 컨슈머 무중단 추가**(ADR-002 §3.3).

### 9.3.5. 보안 (Security)
- 시크릿(DB 비밀번호·API Key·웹훅 URL)은 **절대 커밋 금지**, `.env`/`application-secret.yml`로 분리(행동강령 §5.3.2).
- 인증/인가: 대기열 입장 토큰과 회원 인증을 분리(입장 토큰 ≠ 신원 인증). 상세 정책은 [`10_functional_spec.md`](./10_functional_spec.md).

### 9.3.6. 관측성 (Observability)
- 핵심 지표(대기 인원, 입장률, 재고 잔량, 주문 성공/실패율, p99 지연)를 **메트릭으로 노출**(Actuator/Micrometer).
- 부하 테스트 리포트는 본 문서 §9.2 표를 **실측값으로 갱신**해 추적한다.

### 9.3.7. 유지보수성 (Maintainability)
- 멀티모듈 단방향 의존(아키텍처 §2.2)·ADR 기록·Conventional Commits를 유지한다.

## 9.4. 부하 테스트 계획 (검증 방법)

| 항목 | 내용 |
| :--- | :--- |
| **도구** | k6 / Gatling / nGrinder 중 택1 (Phase 3에서 확정) |
| **시나리오** | §9.1 매트릭스의 대표 셀 B1~B4를 각각 재현 (B2=정합성, B3=처리량, B4=스파이크) |
| **부하 형상** | **스파이크**(`t=0` 전원 동시, 선착순 현실) + **램프업**(점증, 병목 관찰) 2종 |
| **측정 항목** | §9.2 S1~S8 + 자원(CPU·메모리·**DB 커넥션 풀**·Redis ops/s) + 큐 길이·실측 입장률·**완판 소요시간** |
| **산출물** | 부하 테스트 리포트(로드맵 Phase 3) → §9.2 목표값 옆에 **실측값 병기** |
| **합격 기준** | §9.2 모든 SLO 충족, 특히 **S5 오버셀 0건은 무조건 통과** |

### 9.4.1. 측정 시 핵심 고려사항
- **경합비(Contention Ratio)**: 진짜 스트레스 변수는 동시 진입 ÷ 재고. B2(1,000:1)가 락 경합·오버셀 위험의 최악 케이스.
- **결제 전환율 가정**: 예약→결제 성공률 **70%**(나머지 30%는 타임아웃/실패 → 재고 해제 → 재입장). "완판까지 라운드 수"와 재고 회수 흐름(UC-07/P-S4)을 검증.
- **Virtual Threads 효과**: VT vs 플랫폼 스레드풀을 **비교 측정**(처리량·스레드 수·메모리) → 아키텍처 §2.4 전략 1의 정량 증명.
- **워밍업·반복**: JIT 워밍업 후 측정 구간 분리, N회 반복 평균(재현성).
- **환경 고정**: 단일 노드 + 하드웨어 스펙 기록. scale-out 측정은 별도 표기.
- **데이터 시드**: 회원·입장 토큰 사전 준비로 측정 재현성 확보.
