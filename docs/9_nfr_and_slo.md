# 9. 비기능 요구사항(NFR) 및 목표 지표(SLO)

> **Schale Queue**의 정체성은 "대용량 트래픽 제어"다. 기능이 "동작하는가"를 넘어 **"얼마나 잘 버티는가"**를 정량 목표로 고정한다.
> 이 문서의 숫자는 **부하 테스트의 합격/불합격 기준**이며, Phase 3·5의 검증 리포트가 이 표를 직접 채운다.
>
> ⚠️ **상태**: 초안(Draft) — 표의 목표 수치는 **확정 전 제안값**이다. 실측·환경에 맞춰 조정한다.

---

## 9.1. 기준 시나리오 (Reference Scenario)

모든 목표 지표는 아래 **단일 기준 시나리오**를 전제로 측정·해석한다. (수치 확정 필요)

| 변수 | 값(초안) | 의미 |
| :--- | :--- | :--- |
| **한정 재고** | 1,000개 | 단일 인기 상품(Goods)의 Stock |
| **오픈 순간 동시 진입** | 50,000명 | `openAt` 직후 수 초 내 대기열 진입 시도 |
| **유효 구매 의사** | ≈ 재고의 5~50배 | 재고 대비 경합 강도 |
| **측정 환경** | 단일 노드(로컬/단일 서버) 기준, scale-out은 별도 표기 | ADR-001 §3 "단일 DB 기준 이점"과 정합 |

> 이 시나리오는 ADR-001(읽기/쓰기 부하 분리)·ADR-002(전방 ZSET / 후방 Kafka)가 가정한 "선착순 오픈 순간"과 동일하다.

## 9.2. 목표 지표 (SLO) — 정량 합격 기준

| # | 지표 | 목표값(초안) | 측정 지점 | 검증 Phase / 방법 |
| :-- | :--- | :--- | :--- | :--- |
| S1 | **대기열 진입(enqueue) 지연** | p99 < 100ms | `POST /queue/enter` (ZADD) | Phase 3 부하 테스트 |
| S2 | **순번 조회 지연** | p99 < 50ms | `GET /queue/rank` (ZRANK) | Phase 3 부하 테스트 |
| S3 | **입장 처리율(throughput)** | DB 유입 ≤ 500 TPS로 **평탄화** | Worker dequeue rate | Phase 3 — rate 파라미터 |
| S4 | **주문 임계구역 지연** | p99 < 200ms | 주문 생성 + 재고 차감 트랜잭션 | Phase 2/3 |
| S5 | **오버셀(oversell)** | **0건** (재고 1,000 → 정확히 1,000 판매) | Stock 카운터 (sold ≤ total) | Phase 2 동시성 테스트 |
| S6 | **재고 정합성** | `available ≥ 0` **및** `total = available+reserved+sold` 항상 성립 | DB 불변식 (P-S1) | Phase 2 — 동시성 테스트 |
| S7 | **에러율** | 오픈 피크 구간 5xx < 0.1% | API 게이트웨이/액추에이터 | Phase 3 |
| S8 | **주문/결제 이벤트 무유실** | 유실 0건 (at-least-once 이상) | Kafka 컨슈머 오프셋 | Phase 4 |

> **S3 보충**: 입장 처리율은 "느릴수록 안전, 빠를수록 UX 좋음"의 트레이드오프 파라미터다. 목표는 **DB가 견딜 수 있는 상한 이하로 트래픽을 깎는 것**이며, 정확한 값은 S4·S7이 깨지지 않는 최대치를 부하 테스트로 역산한다.

## 9.3. 비기능 요구사항 (NFR) — 품질 속성별

### 9.3.1. 성능 (Performance)
- 대기열 연산(ZADD/ZRANK)은 **메모리 O(log N)** 특성을 유지한다(ADR-002 §3.1).
- 응답 경로의 무거운 작업(알림·정산)은 **사용자 응답 경로에서 분리**한다(후방 Kafka, ADR-002 §2).
- I/O 바운드 구간은 **Virtual Threads**로 처리해 적은 플랫폼 스레드로 고동시성을 확보한다(아키텍처 §2.4 전략 1).

### 9.3.2. 정합성 · 신뢰성 (Consistency / Reliability)
- **재고 초과 판매 0건**은 타협 불가한 최우선 불변식이다(S5/S6).
- 단일 DB: **JPA 비관적 락**(`SELECT ... FOR UPDATE`)으로 Stock 단일 행 직렬화(ADR-001 §2).
- Scale-out: **Redis 분산 락**으로 임계구역을 클러스터 전역 직렬화(아키텍처 §2.4 전략 3).
- 후방 결제 이벤트는 **멱등 처리(idempotency key)** 또는 **트랜잭셔널 아웃박스**로 중복/유실 방어(ADR-002 §3.4).

### 9.3.3. 가용성 · 회복력 (Availability / Resilience)
- **장애 격리**: 전방 대기열 장애가 후방 결제 데이터를 오염시키지 않는다(ADR-002 §3.3).
- **Graceful Degradation**: 대기열 장애 시 신규 입장을 차단하고 재시도를 안내하되, **이미 입장한 주문 흐름은 보호**한다.
- **결제 타임아웃 복원**: 만료된 결제는 워커가 정리하고 **재고를 복원**한다(`Payment.timeoutAt`, ADR-001 §2; 상세는 [`11_domain_policy.md`](./11_domain_policy.md)).

### 9.3.4. 확장성 (Scalability)
- ID 참조 기반 애그리거트 분리로 **도메인별 DB 분리/샤딩 여지**를 확보(ADR-001 §3).
- 후방을 Kafka 이벤트로 풀어 **신규 컨슈머 무중단 추가**(ADR-002 §3.3).

### 9.3.5. 보안 (Security)
- 시크릿(DB 비밀번호·API Key·웹훅 URL)은 **절대 커밋 금지**, `.env`/`application-secret.yml`로 분리(행동강령 §5.3.2).
- 인증/인가: 대기열 입장 토큰과 회원 인증을 분리(입장 토큰 ≠ 신원 인증). 상세 정책은 [`10_functional_spec.md`](./10_functional_spec.md).

### 9.3.6. 관측성 (Observability)
- 핵심 지표(대기 인원, 입장률, 재고 잔량, 주문 성공/실패율, p99 지연)를 **메트릭으로 노출**(Actuator/Micrometer).
- 부하 테스트 리포트는 본 문서 §9.2 표를 **실측값으로 갱신**해 추적한다.

### 9.3.7. 유지보수성 (Maintainability)
- 멀티모듈 단방향 의존(아키텍처 §2.2)·ADR 기록·Conventional Commits를 유지한다.

## 9.4. 부하 테스트 계획 (검증 방법)

| 항목 | 내용(초안) |
| :--- | :--- |
| **도구** | k6 / Gatling / nGrinder 중 택1 (Phase 3에서 확정) |
| **시나리오** | §9.1 기준 시나리오(동시 5만 진입 → 재고 1,000 소진)를 스파이크 부하로 재현 |
| **측정 항목** | §9.2 S1~S7 전부 + 자원 사용률(CPU/메모리/DB 커넥션) |
| **산출물** | 부하 테스트 리포트(로드맵 Phase 3 산출물) → 본 문서 목표값 옆에 **실측값 병기** |
| **합격 기준** | §9.2 표의 모든 SLO 충족, 특히 **S5 오버셀 0건은 무조건 통과** |
