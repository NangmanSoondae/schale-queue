## 2. 여정 요약

```
Phase 1 뼈대(멀티모듈·ERD) → Phase 2 동시성(비관 락, 오버셀 0 증명)
→ Phase 3 대기열(Redis ZSET+SSE, SLO 실측) → Phase 4 EDA(아웃박스·멱등·정산, S8)
→ Phase 5 통합(컨테이너·React·부하·폴리싱) → +AI 어드민(MCP 8툴) → v1.0.0
```

가장 좋았던 리듬: **"슬라이스 구현 → E2E 실증 → 리뷰 → 실측 → 수정"의 반복.** 전체 리뷰 2회가 각각 커다란 결함 수확을 냈고(1회차 H5, 2회차 H2), 부하테스트와 E2E는 리뷰로도 못 잡는 결함을 5건 이상 추가로 적발했다.

## 3. 기술적 교훈 Top 5

1. **정합성 버그는 대부분 '시점'의 문제였다.** 최악의 결함(P-O3 한도 우회)은 REPEATABLE READ 스냅샷 읽기와 최신 커밋 읽기(잠금/유니크)가 **서로 다른 시점을 보호**해서 생겼다. "검사와 강제가 같은 시점을 보는가"가 동시성 설계의 제1질문. 같은 부류: 결제 만료 타임존(No.10), 시간대 3원화.
2. **원자성은 재시도가 아니라 설계로 산다.** 부분 실패로 생기는 중간 상태(고아 큐, 토큰 없는 이탈자)는 재시도로 복구가 안 된다 — Redis 멀티스텝은 Lua 하나로, DB 변경+이벤트 발행은 트랜잭셔널 아웃박스로 묶는 것이 유일한 답이었다.
3. **측정이 직관을 이긴다.** SSE 직렬 브로드캐스트는 실측 전까지 "충분히 빠를 것"이었다 — 실측은 2,000구독에서 주기 포화를 보였고, 벌크화로 12.4×를 얻었다. VT도 마찬가지: "빠르다"가 아니라 "스레드 36 vs 216, 느린 블로킹에서만 3.1×"라는 조건부 진실을 얻었다.
4. **게이트형 테스트의 그린은 두 번 의심하라.** env 게이트 IT는 스킵돼도 그린이다(No.15 — Gradle 데몬이 env를 삼킴). exit code를 파이프가 삼키면 CI 게이트도 장식이 된다(No.14 — 실제 레드 머지 사고). "통과한 그린"과 "실행되지 않은 그린"을 구분하는 절차(skipped=0 확인, exit code 단독 판정)를 사고 후에야 갖췄다.
5. **E2E는 실제 장애를 일으켜야 완성된다.** SSE 재연결 로직 검증에서 api를 '실제로 재시작'했더니, 단위 테스트로는 영원히 못 잡을 nginx DNS 고정 결함(No.16)이 나왔다. 재시도/재연결 코드의 테스트는 의존 서비스를 진짜로 죽였다 살려야 한다.

## 4. 프로세스 교훈

- **CI 그린 = 머지 게이트의 자율 개발**은 사람 승인 없이도 3주간 43 PR을 안전하게 흘렸다. 단 한 번의 레드 머지 사고(No.14)도 게이트 '구현'의 결함이었지 원칙의 결함이 아니었고, 그 사고가 절차를 더 단단하게 만들었다.
- **트러블슈팅 5단계 누적은 복리로 돌아온다.** No.05(스키마 불일치)의 교훈이 M6·V5 설계에, No.12(dash vs bash)가 워커 헬스체크에, #25(ObjectMapper)가 admin-mcp에서 그대로 재사용됐다 — "같은 실수를 두 번 하지 않는 자산"이 실제로 작동했다.
- **AI 리뷰 에이전트의 제안은 검증이 필수다.** 리뷰 2회차의 H-1 지적은 정확했지만 제안한 수정(슬롯 선INSERT)은 FK 제약상 불가능했다 — 발견과 처방은 별개의 신뢰도를 가진다. 교차 검증(반박 시도) 단계가 있었기에 잡혔다.
- **정직한 기록이 리포트의 가치를 만든다.** "N=5,000 실측은 하네스 한계로 무산", "이 수치는 Docker NAT가 지배" 같은 한계 명시는 리포트의 신뢰도를 깎는 게 아니라 세운다.
