## §9.2 SLO 실측 (배포 구성 = Virtual Threads ON)

| # | 지표 | 목표 | 실측(VT on) | 판정 |
|---|------|------|------|------|
| S1 | enqueue p99 | < 100ms | **21.66ms** | ✅ PASS |
| S2 | position p99 | < 50ms | **13.01ms** | ✅ PASS |
| S4 | 주문 임계구역 p99 (저동시성 = 입장률 조절 재현) | < 200ms | **106.59ms** | ✅ PASS |
| S5 | 오버셀 | 0건 | **0건** (성공 10 / 충돌 990) | ✅ PASS |
| S6 | 재고 정합성 total = sold + remain | 성립 | **10 = 10 + 0** | ✅ PASS |
| S7 | 5xx 에러 (B2) | 0건 | **0건** | ✅ PASS |

> **S4 보충(§9.2 "B1 충족·B2 추적")**: 입장 토큰을 직접 시드해 **200-way 버스트**(큐 스로틀 우회)를 임계구역에 한꺼번에 던진 B2 극한에서는 주문 p99가 콜드 411ms → 워밍업 후 ~76ms로 측정됐다. 비관적 락이 단일 재고 행을 직렬화하므로 동시성이 높을수록 열화하는 것은 **예상된 추적 대상**이며, 현실에서는 워커가 입장률(S3 ≤ 200~500 TPS)로 동시성을 깎아 저동시성 경로(106.59ms)에 가깝게 동작한다.

### B1 처리량
- enqueue+position 왕복 **~1,728 iter/s (~3,457 req/s)**, 207,396 요청 중 실패 0.02%(종료 구간 stragglers).

## VT vs 플랫폼 스레드 비교 (§9.4.1)

동일 시나리오를 `spring.threads.virtual.enabled` on/off 로 각 1회 측정(단일 머신·단일 실행, **지표성**).

| 지표 | Virtual Threads ON | 플랫폼 스레드 | 비고 |
|------|-----------------|---------------|------|
| B1 enqueue p99 | 21.66ms | 26.55ms | 동등 수준 |
| B1 position p99 | 13.01ms | 15.81ms | 동등 수준 |
| B1 처리량(iter/s) | 1,728 | 1,825 | 오차 범위(±5%) |
| S4 주문 p99(저동시성) | 106.59ms | 226.13ms* | VT 우세하나 *꼬리 스파이크(max 294ms) 영향, 단일 실행 |
| 스레드/메모리 peak | 미계측 | 미계측 | 후속: Actuator/Micrometer(§9.3.6) |

### 해석
- **모든 대상 SLO는 배포 구성(VT on)에서 충족**된다.
- 테스트 동시성(100~200 VU)에서 VT와 플랫폼 스레드는 **지연·처리량이 대체로 동등**하다. 이 구간의 병목은 스레드 모델이 아니라 **단일 재고 행의 비관적 락·DB 커넥션 풀·Redis 단일 커넥션**이기 때문이다.
- VT의 본질적 이점은 **수천 개의 블로킹 I/O 연결**(장수명 SSE 구독 등)에서 드러난다 — 본 1차 측정은 그 규모를 밀어붙이지 않았다(후속 과제).

## 결론

- §9.2 대상 SLO(S1/S2/S4/S5/S6/S7) **전부 충족** ✅. 특히 **오버셀 0건(S5/S6)은 200-way 경합에서도 불변식 성립**.
- 부하테스트 중 **첫 실 DB 부팅에서 잠복 스키마 불일치(goods.description)를 발견·수정**(troubleshooting No.05).
- VT는 이 동시성 구간에서 플랫폼 스레드와 동등하며, 고연결 SSE 부하에서의 정량 우위 증명은 후속.
